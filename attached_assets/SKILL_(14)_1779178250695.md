# Adding a Public Data Source to Auralyn

After the I001 framework is in place (`server/ingestion/sources/`), every
new source follows this pattern.

## Decision: is this source appropriate?

**Yes:**
- US federal agencies (CDC, FDA, NIH, USPSTF, AHRQ, CMS) — public
  domain, free APIs, citable
- State health department open data portals — usually public domain
- WHO disease outbreak news — free, citable
- NLM resources (DailyMed, RxNorm, MeSH, PubMed) — free, structured

**No (do not ingest):**
- UpToDate, DynaMed, Epocrates premium — paid, copyrighted, ToS
  prohibits scraping
- Anything behind a login wall — CFAA risk
- Anything that requires HTML parsing (sign you picked the wrong source)
- Sources without stable APIs or version tracking

## The interface to implement

```typescript
// server/ingestion/sources/<source>.ts
import { PublicDataSource } from "./types";
import { fetchClient } from "./fetchClient";

export const <sourceName>: PublicDataSource = {
  id: "<source-id>",           // stable string, used in audit logs
  name: "<Human Name>",
  baseUrl: "<api-base-url>",
  rateLimit: { requests: 240, perSeconds: 60 },  // source-specific
  auth: undefined,             // or { type: "api_key", header: "..." }

  async fetch(query) {
    return fetchClient.get(`${this.baseUrl}/...`, {
      source: this.id,
      params: query,
    });
  },

  normalize(raw): MemoryEntryDraft[] {
    // Map raw payload → array of clinical_memory entries
    return raw.results.map(item => ({
      key: `<prefix>:<sub-prefix>:<unique-id>`,
      scope: "global",
      content: <human-readable summary>,
      confidence: 0.95,         // verified external guideline
      verifiedBy: "external_guideline",
      source: `${this.name} ${item.version || item.date}`,
    }));
  },
};
```

## The scheduled job pattern

```typescript
// server/jobs/<source>Sync.ts
import { Queue, Worker } from "bullmq";
import { <sourceName> } from "../ingestion/sources/<source>";
import { writeGlobalGuideline } from "../context/memoryWriters";

const QUEUE_NAME = "<source>-sync";

// Schedule: pick cadence appropriate to source update frequency
// - Daily for FDA recalls
// - Weekly for CDC surveillance, USPSTF
// - On-demand for DailyMed (per-drug lookups)
export const <source>SyncQueue = new Queue(QUEUE_NAME, { connection });

export async function scheduleSync() {
  await <source>SyncQueue.add(
    "scheduled",
    {},
    { repeat: { pattern: "0 4 * * *" } }  // 04:00 UTC daily — cron syntax
  );
}

new Worker(QUEUE_NAME, async (job) => {
  const raw = await <sourceName>.fetch(job.data);
  const entries = <sourceName>.normalize(raw);

  let accepted = 0, rejected = 0;
  for (const entry of entries) {
    const result = await writeGlobalGuideline(entry);
    if (result.accepted) accepted++; else rejected++;
  }

  emitTelemetry("auralyn.ingestion.sync_complete", {
    source: <sourceName>.id,
    fetched: entries.length,
    accepted,
    rejected,
  });
}, { connection });
```

## Key naming convention for clinical_memory

Use hierarchical keys so `keyPrefix` searches work:

```
<category>:<subcategory>:<unique-id>
```

Established prefixes:
- `surveillance:respiratory:<state>:<week>` — CDC FluView etc.
- `safety:drug_recall:<rxcui>:<recall_id>` — openFDA recalls
- `safety:drug_alert:<alert_id>` — openFDA safety communications
- `labeling:drug:<rxcui>` — DailyMed SPL
- `preventive:uspstf:<topic_id>` — USPSTF recommendations
- `guideline:<society>:<id>` — published professional society guidelines

When inventing a new prefix, document it in this skill file so future
sources follow the same pattern.

## Conflict resolution (automatic via ClinicalMemoryStore)

Same `key` + same `scope` triggers conflict resolution:
- Newer entry with higher-or-equal confidence + verification wins
- Incumbent is revoked (not deleted — preserves audit)
- Lower-confidence updates are rejected

This means re-running the same sync produces no duplicates as long as
the source's `version` or `date` field is captured in the key.

## Verification (every new source)

After implementing:

```bash
# 1. Run the sync manually once
npx ts-node server/jobs/<source>Sync.ts

# 2. Confirm rows written
psql -c "SELECT count(*), max(created_at)
FROM clinical_memory
WHERE source LIKE '%<source name>%' AND status='active';"

# 3. Confirm audit log captured the fetch
psql -c "SELECT source_id, http_status, duration_ms
FROM ingestion_audit
WHERE source_id='<source-id>'
ORDER BY fetched_at DESC LIMIT 5;"

# 4. Run a real encounter and confirm the data surfaces
```

## Read-path wiring

Inserting rows is not enough — `fetchLearnedContext` in
`server/context/memoryWriters.ts` must include the new key prefix.
Without this, entries are written but never surface in agent prompts.

```typescript
// In fetchLearnedContext, add the new prefix:
const surveillanceMatches = await memoryStore.retrieve({
  scope: { tenantId, physicianId: undefined },  // global lookups
  keyPrefix: "surveillance:",
});
```

## Hard rules

1. **No HTML parsing.** If the source requires it, you picked the wrong
   source.
2. **No PHI in ingested entries.** Public data only.
3. **Every entry has a `source` field with a citable reference.**
4. **The audit log captures every fetch** (success or failure).
5. **Idempotent sync.** Re-running the same sync produces no duplicates.
6. **Rate limit respect.** Use the source's documented rate limit;
   `fetchClient` enforces it via backoff.
