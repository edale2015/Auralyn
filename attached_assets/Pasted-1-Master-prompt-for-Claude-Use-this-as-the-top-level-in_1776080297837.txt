Use this as the top-level instruction before you start feeding code.

Writing

You are performing a rigorous code review of a production-oriented medical triage and urgent care software platform.

Your job is not to praise the architecture or restate what the system intends to do. Your job is to identify concrete weaknesses, defects, unsafe assumptions, misleading scaffolding, incomplete integrations, hidden production risks, and mismatches between UI, backend, and actual runtime behavior.

Review the code as if this system may eventually support real clinical workflows, physician oversight, auditability, regulated deployment, and multi-tenant production use.

Important review rules:

Be skeptical
Do not assume a feature is real just because routes, UI, or comments exist
Trace data flow when possible
Prefer findings tied to actual files and code behavior
Call out uncertain findings clearly as “needs confirmation”
Distinguish between “bad style” and “real risk”

2. Best review sequence

Do not feed the repo randomly. Review by system slice. That catches the snakes in the pipes.

Use this order:

Batch 1: Auth, RBAC, audit, governance

Include files like:

auth middleware
JWT/session handling
role enforcement
audit logger
hash-chain logic
override approval flows
governance endpoints

Ask Claude:

Writing

Review these files only for auth, authorization, audit integrity, governance enforcement, override control, and tenant isolation. Ignore style. Find concrete security, compliance, and enforcement risks.

Batch 2: Intake, reasoning, red flags, disposition

Include:

complaint intake
modifier intake
red flag detection
scoring engines
Bayesian engine
deterministic rules
disposition engine
treatment suggestion logic

Ask:

Writing

Review these files only for clinical safety, contradiction risk, missing escalation logic, inconsistent reasoning, unsafe fallbacks, and hidden disagreement between engines.

Batch 3: KB, Postgres, cache, migrations

Include:

KB CRUD/services
schema
migrations
cache layer
sync/import code
feature model loading
fallback logic

Ask:

Writing

Review these files only for source-of-truth integrity, stale logic risk, migration safety, cache incoherence, hardcoded fallbacks, and places where KB-driven behavior is undermined.

Batch 4: Dashboards, APIs, WebSockets

Include:

dashboard pages/components
API routes
route handlers
websocket stream code
health/status panels
live patient views

Ask:

Writing

Review these files only for frontend/backend mismatches, fake controls, misleading status indicators, broken contracts, incomplete live wiring, and dashboard behavior that implies functionality not truly enforced by the backend.

Batch 5: EHR, FHIR, automation adapters

Include:

Epic/ECW/Athena code
SMART on FHIR
UI automation
selector healing
write-back endpoints
external adapter services

Ask:

Writing

Review these files only for realism of integration, failure safety, write-back risk, incomplete adapter paths, insecure automation behavior, and features presented as real but only partially implemented.

Batch 6: Monitoring, RLHF, learning, simulation

Include:

monitoring services
drift detection
RLHF or weight tuning
eval harness
golden cases
simulation engines
autonomous loops

Ask:

Writing

Review these files only for production safety, false claims of self-improvement, unsafe autonomous behavior, weak monitoring assumptions, poor validation logic, and inability to prove correctness.

4. How to split code for Claude

Best options:

Good
one subsystem at a time
all related files together
include routes + services + UI for that subsystem
Bad
random 20-file chunks
only frontend without matching backend
only interfaces without implementation
only high-level descriptions

A good chunk is usually:

8 to 20 related files
enough to trace behavior
not so huge that review becomes mush
5. What to ask Claude to avoid

Tell Claude not to waste time on:

formatting preferences
naming nits
generic “consider adding tests”
vague “could be more modular”
speculative praise
architecture poetry with no file evidence

You want scalpels, not confetti.
