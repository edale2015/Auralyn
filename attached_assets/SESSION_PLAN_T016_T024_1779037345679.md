## T016 — Full artifact publication coverage across all agents

**Status:** ⏳ Not started

**Problem this closes:** Currently only 2 artifact-bus publish points exist
(steps 6 and 9). Every other agent step is passing reasoning forward via
transcript, which is exactly the failure mode the article describes.

**Files to modify:** every agent file in `server/clinical/agents/` (or
wherever the 13-step pipeline lives).

**Required publication matrix** — every row must be implemented:

| Pipeline step | Agent role | Artifact type | When to publish |
|---|---|---|---|
| 1 | triage | `validated_finding` | one per intake field actually captured (CC, vitals, allergies, meds, PMH) |
| 2 | triage | `validated_finding` | red-flag rule result, positive OR negative — record both |
| 3 | triage | `validated_finding` | acuity classification |
| 4 | triage | `validated_finding` | each finding extracted from Q&A round 1 |
| 5 | differential | `kb_retrieval` | each KB chunk consulted, with citation |
| 6 | differential | `ruled_out` | each diagnosis dropped from initial differential, with reason |
| 6 | differential | `uncertainty` | each unresolved question that blocks ranking |
| 7 | differential | `calculation` | each risk score computed (HEART, Wells, PERC, NEXUS, etc.) |
| 8 | differential | `validated_finding` | each finding from Q&A round 2 |
| 9 | differential | `ruled_out` | each diagnosis eliminated after Q&A round 2 |
| 10 | disposition | `kb_retrieval` | each protocol or guideline consulted |
| 11 | disposition | `decision` | each candidate disposition proposed (NOT just the final) |
| 12 | disposition | `decision` | the final disposition |
| 12 | disposition | `uncertainty` | each unresolved question affecting disposition |
| 13 | billing | `decision` | each CPT/E&M code proposed |
| 13 | billing | `uncertainty` | each documentation gap that would block a clean claim |
| any | any | `failed_attempt` | every tool call that returned empty or unusable result |
| any | supervisor | `ruled_out` / `decision` | every supervisor intervention |

**Acceptance criteria:**
- Every row above is implemented — locate each agent step and add the
  `bus.publish(role, artifact)` call
- Each artifact carries a `provenance` object with non-empty `source`
  field and either `kbChunkIds`, `toolCallId`, or `citation`
- A realistic chest-pain encounter through all 13 steps produces **at
  least 15 artifacts**, of at least 4 distinct types
- Bus contract violations throw an error (do not swallow) — keep the
  existing behavior from T010

**Verification:**
```bash
# Run the chest-pain example encounter
npx ts-node server/context/example_encounter.ts > /tmp/encounter.log 2>&1

# Count artifacts published
grep "bus.publish\|publish(" /tmp/encounter.log | wc -l
# expect: ≥ 15

# Count distinct artifact types in the encounter
curl -s http://localhost:3000/api/context/$(...)/state | \
  jq '[.artifacts[].type] | unique | length'
# expect: ≥ 4

# Confirm provenance is populated on every artifact
curl -s http://localhost:3000/api/context/$(...)/state | \
  jq '[.artifacts[] | select(.provenance.source == null or .provenance.source == "")] | length'
# expect: 0
```

**Dependencies:** T006, T008, T009, T010

**Do not skip any row in the matrix.** If a pipeline step doesn't
currently produce information that fits one of these artifact types, that
itself is a finding — report it; do not silently omit the publication.

## T017 — Compactor check before EVERY agent step

**Status:** ⏳ Not started

**Problem this closes:** Compactor currently fires only once, before step
12. Long or complex encounters can accumulate enough working context to
degrade reasoning well before that.

**Files to modify:**
- Pipeline orchestrator (the file that calls the 13 steps in sequence)

**Acceptance criteria:**
- Before EVERY agent step (1 through 13), call
  `compactor.shouldCompact(ctx)` and run `compactor.compact(ctx)` if true
- Each compaction event is logged via the existing `appendCompactionEvent`
  function and persisted to the inspector cache
- Compaction events include: pre-step number, before/after tokens,
  artifacts emitted by the compactor
- A 25-turn simulated encounter must trigger compaction **at least twice**
  without losing any red flag or hard constraint

**Verification:**

## T018 — Verify artifact CONSUMPTION at every agent step

**Status:** ⏳ Not started

**Problem this closes:** Publishing without consuming is an expensive
no-op. We must prove downstream agents are actually reading from the bus
and using those artifacts in their prompts.

**Files to modify:**
- Every agent file from T016
- `server/context/ClinicalContextManager.ts` (if assemblePromptFor doesn't
  already pull from a bus reference)

**Acceptance criteria:**
- Before each agent's model call, the agent calls `bus.readFor(role)` and
  the returned artifacts are passed to / available to `assemblePromptFor`
- The prompt assembly includes a `## DURABLE ARTIFACTS` section that lists
  artifact ids it included (use the existing `includedArtifactIds` field
  on `AssembledPrompt`)
- The billing agent's prompt at step 13 contains STRICTLY FEWER artifacts
  than the differential agent's prompt at step 9 (enforced by the
  consume-contract in `AgentArtifactBus`, but verify it actually holds in
  practice)
- For each agent step, log: `[role] consumed_artifacts=N, prompt_tokens=M`

**Verification:**
```bash

npx ts-node server/context/example_encounter.ts > /tmp/enc.log
grep "consumed_artifacts" /tmp/enc.log

# Pull prompt previews and compare artifact counts

  jq '.includedArtifactIds | length' > /tmp/diff.count

curl -s http://localhost:3000/api/context/$ID/prompts/billing | \
  jq '.includedArtifactIds | length' > /tmp/bill.count
```

## T019 — Memory write paths (preferences, protocols, RLHF deltas)

**Status:** ⏳ Not started

**Problem this closes:** The `clinical_memory` table exists and the
demotion sweep runs, but nothing is currently writing to it. With no
writes, the cross-encounter learning the article describes does not
happen, and the demotion sweep operates on an empty table.

**Files to create:**
- `server/context/memoryWriters.ts` — concrete write functions per memory
  category

**Files to modify:**
- `server/clinical/agents/supervisorAgent.ts` (or wherever supervisor
  overrides are handled) — call the writer when an override happens
- Tenant admin UI backend route (wherever protocols are added) — call the
  writer when a protocol is added
- `server/clinical/agents/differentialAgent.ts` and dispositionAgent —
  read from memory store when assembling reasoning context

**Write paths to implement:**

| Trigger | Memory key pattern | Scope | Source |
|---|---|---|---|
| Supervisor overrides a disposition | `rlhf:disposition:<complaint_id>:<from>→<to>` | physician | `verifiedBy: "supervisor"` |
| Supervisor adds a hard constraint | `constraint:<encounter_complaint>:<constraint_slug>` | physician (or tenant if marked) | `verifiedBy: "supervisor"` |
| Admin adds tenant protocol | `protocol:<protocol_id>` | tenant | `verifiedBy: "external_guideline"` |
| Global KB updates a guideline | `guideline:<guideline_id>` | global | `verifiedBy: "external_guideline"` |

**Read path:**
- When `ClinicalContextManager.assemblePromptFor(role, ...)` runs, query
  the memory store with the encounter's tenant + physician scope and
  prepend relevant entries to the working context block (under a new
  `## LEARNED PREFERENCES & PROTOCOLS` section)
- Apply scope precedence: physician > tenant > global (already in the
  store; just call `memoryStore.retrieve()` and use the result)

**Acceptance criteria:**
- Each write trigger above produces a row in `clinical_memory` —
  verifiable via SQL
- The demotion sweep, run after a few weeks of writes, actually demotes
  unused entries (not a no-op anymore)
- The prompt preview endpoint shows learned preferences when relevant
  ones exist for the scope

**Verification:**
```bash
# Simulate a supervisor override and confirm a memory row is written
curl -X POST http://localhost:3000/api/encounter/$ID/supervisor-override \
  -d '{"original":"home_with_rx","override":"ed_transfer","reason":"red flag missed"}'

psql -c "SELECT count(*) FROM clinical_memory WHERE key LIKE 'rlhf:disposition:%';"
# expect: ≥ 1

# Confirm read path surfaces the preference in the next encounter prompt
curl -s http://localhost:3000/api/context/$NEW_ID/prompts/disposition | \
  jq '.userPrompt' | grep -c "LEARNED PREFERENCES"
# expect: 1
```

## T023 — KV-cache prefix stability verification

**Status:** ⏳ Not started

**Problem this closes:** The article explicitly calls out the KV-cache
penalty. Stable per-role prompt prefixes are what let caching actually
help. We need to verify this holds in practice.

**Files to modify:**
- `ClinicalContextManager.ts` — add a `getPromptPrefixHash(role)` method
  that hashes the system prompt + immutables block (the parts that
  SHOULD be stable across calls within an encounter)

**Files to create:**
- `tests/integration/prefix_stability.test.ts`

**Acceptance criteria:**
- `getPromptPrefixHash(role)` returns a deterministic hash of the stable
  prefix for a role
- The hash is stable across consecutive calls within an encounter UNLESS
  immutables have changed (red flag added, hard constraint added)
- Test asserts: in a 13-step encounter where immutables change exactly
  once (at step 2 when red flag fires), each role's prefix hash takes
  at most 2 distinct values across the whole encounter
- Telemetry from T020 emits `auralyn.context.prefix_stability` (1 if hash
  unchanged from prior call, 0 if changed) and the daily aggregate p50
  should be ≥ 0.85 (i.e., 85% of consecutive role-calls hit the same
  prefix)

**Verification:**
```bash
npm test -- tests/integration/prefix_stability.test.ts
# expect: pass

# Check production aggregate
psql -c "SELECT metric_value_p50 FROM context_metrics_daily WHERE metric_name='auralyn.context.prefix_stability' AND metric_date = CURRENT_DATE - 1;"
```
