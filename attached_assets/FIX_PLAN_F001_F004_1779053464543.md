## F001 — Database fallback for `loadComplaintConfig` (CRITICAL)

**Status:** ⏳ Not started

**Files to modify:**
- `server/clinical/loadComplaintConfig.ts` (or wherever `loadComplaintConfig`
  is defined — Replit assistant: find the file that exports this function)
- `server/clinical/unifiedClinicalPipeline.ts` — line 295 area, ensure the
  pipeline does NOT exit when Sheets fails but a DB-derived config exists

**Files to create:**
- `server/clinical/loadComplaintConfigFromDB.ts` — reads from
  `kb_master_rules` and returns the same shape as the Sheets loader

**Acceptance criteria:**
1. `loadComplaintConfig(complaintId)` tries Sheets first; on any
   failure (network error, auth error, quota error, parse error), falls
   back to `loadComplaintConfigFromDB(complaintId)`.
2. `loadComplaintConfigFromDB` returns a config object with the EXACT same
   shape Sheets returns (Replit assistant: read the Sheets loader to learn
   the shape; then construct the same shape from `kb_master_rules` rows
   joined and grouped by `complaint_id` and `step_number`).
3. When the DB fallback is used, `staleConfig: true` is set on the result
   (so downstream code knows the source) AND a telemetry event
   `auralyn.context.config_fallback_used` is emitted.
4. If BOTH Sheets and DB fail (or DB returns zero rows for the complaint),
   THEN return `buildSafeDefault` as today.
5. NO change to the function signature. NO new mandatory parameters. All
   existing call sites continue to work without modification.

**Verification:**

  | jq -r '.sessionId')

curl -s localhost:3000/api/context/$ENCOUNTER_ID/state | jq '{
  total: (.artifacts | length),
  types: ([.artifacts[].type] | unique),
  stale_config: .staleConfig

**Required output:**
- `total ≥ 12`
- `types` array length ≥ 4
- `stale_config: true` (because DB fallback was used)
