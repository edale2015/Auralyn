## V001 — Real-KB parity (not the demo config)

**Purpose:** T016/T018 numbers were collected against the inline demo
config (where `WHEN_EXPR: "true"` forces every workup row to fire). This
verifies the same plumbing produces healthy numbers against the live
`kb_master_rules` table.

**Run this:**
```bash
# Real-KB chest pain encounter (NOT /api/encounter/demo)
ENCOUNTER_ID=$(curl -s -X POST localhost:3000/api/encounter \
  -H "Content-Type: application/json" \
  -d '{
    "chief_complaint": "chest pain",
    "age": 58,
    "sex": "M",
    "allergies": ["NKDA"],
    "medications": ["lisinopril 20mg qd", "atorvastatin 40mg qd"],
    "pmh": ["HTN","Hyperlipidemia","Former smoker"],
    "vitals": {"hr":96,"sbp":158,"dbp":92,"rr":18,"spo2":97,"temp_c":36.8,"pain":6}
  }' | jq -r '.encounter_id')

curl -s localhost:3000/api/context/$ENCOUNTER_ID/state | jq '{
  total: (.artifacts | length),
  types: ([.artifacts[].type] | unique),
  type_count: ([.artifacts[].type] | unique | length),
  by_type: ([.artifacts[].type] | group_by(.) | map({type: .[0], count: length})),
  red_flags: (.immutables.redFlagsIdentified | length),
  hard_constraints: (.immutables.hardConstraints | length)
}'
```

**Report:** paste the JSON output.

**Healthy:** `total ≥ 12` (allowing some drop from the 24 in demo since
real KB is more selective), `type_count ≥ 4`, `red_flags ≥ 1` for this
chest-pain presentation in a 58yo M with cardiac risk factors.

**Unhealthy:** if `total < 8` or `type_count < 3`, the real KB isn't
producing enough artifact diversity — that means a lot of clinical
reasoning is still happening implicitly inside prompts rather than being
externalized as artifacts. Report which artifact types are missing.

## V002 — Per-role artifact breakdown on a real encounter

**Purpose:** T018 passed at 17-vs-18 but that's a narrow delta. On a real
encounter, billing should see meaningfully fewer artifacts than
differential, because billing's consume contract is only
`validated_finding` + `decision` while differential's covers 6 types.

**Run this (against the same `$ENCOUNTER_ID` from V001):**
```bash
for ROLE in triage differential disposition billing supervisor; do
  RESULT=$(curl -s localhost:3000/api/context/$ENCOUNTER_ID/prompts/$ROLE)
  COUNT=$(echo "$RESULT" | jq '.includedArtifactIds | length')
  TOKENS=$(echo "$RESULT" | jq '.estimatedTokens')
  echo "$ROLE: artifacts=$COUNT tokens=$TOKENS"
done
```

**Report:** paste the 5 lines.

**Healthy:** monotonically reasonable — triage smallest (≤3), differential
largest, billing < differential by at least 3, supervisor sees the most.
Example healthy output:
```
triage: artifacts=2 tokens=820
differential: artifacts=14 tokens=4200
disposition: artifacts=11 tokens=3800
billing: artifacts=6 tokens=2100
supervisor: artifacts=15 tokens=5200
```

**Unhealthy:** if billing ≥ differential, or any role's tokens exceed its
budget in `PROMPT_BUDGET_TOKENS`, the filtering isn't doing its job.

## V004 — Daily aggregate population (defer 48 hours)

**Purpose:** Confirm `contextMetricsAggregate.ts` actually populates
`context_metrics_daily` after real traffic has accumulated.

**When to run:** at least 48 hours after V001's encounter and other real
traffic.

**Run this:**
```bash
psql -c "SELECT
  metric_name,
  count(*) as days_with_data,
  max(metric_date) as latest_date,
  avg(metric_value_p50)::numeric(10,2) as avg_p50,
  avg(metric_value_p95)::numeric(10,2) as avg_p95
FROM context_metrics_daily
WHERE metric_date >= CURRENT_DATE - INTERVAL '7 days'
GROUP BY metric_name
ORDER BY metric_name;"
```

**Report:** paste the table.

**Healthy:** at least 5 rows, one per metric type
(`auralyn.context.prompt_tokens`, `artifacts_published`,
`artifacts_excluded_for_budget`, `compaction_event`, `prefix_stability`,
`memory_hits`, `bus_contract_violation`). `days_with_data ≥ 1` for each.

**Unhealthy:** empty result = the scheduled job isn't running, or it's
running but failing silently. Check server startup logs for
`scheduleContextMetricsAggregate` and any error from the job itself.

## V005 — Demotion sweep end-to-end (defer 4 weeks)

**Purpose:** Confirm the demotion sweep actually moves stale entries
through `active → shadow → revoked` against real aged data.

**When to run:** at least 4 weeks after T019 memory writes start
happening in production.

**Run this:**
```bash
psql -c "SELECT
  status,
  scope,
  count(*) as entries,
  min(created_at) as oldest,
  max(last_retrieved) as last_used
FROM clinical_memory
GROUP BY status, scope
ORDER BY status, scope;"

# Also confirm the sweep job ran recently
psql -c "SELECT * FROM job_run_log
WHERE job_name = 'memoryDemotionSweep'
ORDER BY run_started DESC LIMIT 5;"
```

**Report:** paste both outputs.

**Healthy:** non-zero counts in `shadow` status by week 4 (unused entries
being demoted). Sweep job log shows daily runs with no failures.

**Unhealthy:** all entries still `active` after 4 weeks means either
nothing is unused (unlikely) or the demotion thresholds in
`DEFAULT_DEMOTION_POLICY` are too loose for the actual usage pattern.
Worth tuning at that point.
