B) Cloud Storage migration (keep API contract identical)

Right now: files go to local disk. That’s fine for single instance, but not for production multi-instance. We’ll change only the storage backend in routes.files.ts and the file metadata store.

B1) Decide: Firebase Storage or direct GCS?

Since you’re already using Firestore, go with Firebase Storage (it is Google Cloud Storage under the hood, with nicer Firebase tooling). Use the Firebase Admin SDK.

B2) Metadata schema (minimal change)

Your current files record probably has:

fileId

token

mimeType

storagePath (local)

Change to:

storageMode: "local_disk" | "firebase_storage"

bucket

objectPath

keep mimeType, originalName, createdAt

keep token for ownership checks

Example metadata for cloud:

{
  fileId,
  token,
  originalName,
  mimeType,
  createdAt,
  storageMode: "firebase_storage",
  bucket: "your-bucket.appspot.com",
  objectPath: `intake_uploads/${token}/${fileId}`
}

B3) Upload implementation (server)

In your server/firebase.ts (or shared init), ensure you initialize admin storage too:

import admin from "firebase-admin";

admin.initializeApp({
  projectId: process.env.FIREBASE_PROJECT_ID,
  storageBucket: process.env.FIREBASE_STORAGE_BUCKET // optional; else default
});


Set env (recommended explicitly):

FIREBASE_STORAGE_BUCKET=<your-bucket>.appspot.com
UPLOADS_MODE=firebase_storage

Upload route change (core idea)

Instead of writing to UPLOAD_DIR, do:

const bucket = admin.storage().bucket();
const objectPath = `intake_uploads/${token}/${fileId}${ext}`;

await bucket.file(objectPath).save(file.buffer, {
  contentType: file.mimetype,
  resumable: false,
  metadata: {
    cacheControl: "private, max-age=0"
  }
});


Then write file metadata via your store (Firestore or SQLite). Same response to the client.

B4) Download implementation (keep PHI access server-controlled)

Two safe patterns:

Pattern 1: Stream through your server (simplest + safest)

Check ownership (token verified OR provider key)

Fetch object from bucket

Pipe to response

const f = bucket.file(objectPath);
res.setHeader("Content-Type", meta.mimeType);
f.createReadStream().pipe(res);

Pattern 2: Signed URL (scales better, still safe)

Validate ownership first

Then generate a signed URL valid for ~60 seconds

Redirect client to it or return it

For PHI, Pattern 1 is often cleaner because your server remains the gate.

B5) Backward compatibility

Keep supporting old local-disk uploads for any existing records:

if storageMode === "local_disk" → serve from filesystem

else → stream from bucket

This lets you migrate gradually.
