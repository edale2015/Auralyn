# AURALYN SESSION PLAN — PUBLIC DATA INGESTION (I001–I006)

**Continuing from F001–F004 (production gap closure — ✅ Complete)**
**This plan: 6 ingestion tasks (I001–I006)**

**Goal:** Replace ad-hoc clinical content gathering with proper public-API
ingestion. All sources in this plan are public domain or have explicit
programmatic APIs — no scraping, no ToS risk, all auditable for the 510(k)
submission. Ingested content writes to `clinical_memory` (global scope)
via `writeGlobalGuideline`, then flows through the existing memory read
path in `ClinicalContextManager.assemblePromptFor`.

**Architecture principle:** No source touches the clinical pipeline
directly. All flow is:

```
Scheduled BullMQ job  →  fetch from public API  →  normalize  →
writeGlobalGuideline  →  clinical_memory (global)  →
ContextManager retrieve (per-encounter)  →  agent prompt
```

This keeps human review in the loop (you see new entries in the Context
Health dashboard) and produces a clean audit trail.

## I001 — Public-source ingestion framework

**Status:** ⏳ Not started

**Files to create:**
- `server/ingestion/sources/types.ts` — `PublicDataSource` interface
- `server/ingestion/sources/fetchClient.ts` — HTTP client with retry,
  rate limit handling, timeout, and audit logging
- `server/ingestion/normalize.ts` — utilities to map source-specific
  payloads into the `MemoryEntry` shape `writeGlobalGuideline` expects
- `server/ingestion/auditLog.ts` — every fetch is logged with source,
  URL, request time, response status, and a hash of the payload

**Acceptance criteria:**
1. `PublicDataSource` interface has fields: `id`, `name`, `baseUrl`,
   `rateLimit`, `auth?`, `fetch(query): Promise<RawPayload>`,
   `normalize(raw): MemoryEntryDraft[]`
2. `fetchClient` handles 429 rate limit with exponential backoff,
   timeouts at 30s, and emits telemetry event
   `auralyn.ingestion.fetch` per call
3. Every fetch produces an audit log entry written to a new
   `ingestion_audit` table (DDL in this task)
4. No source-specific code in the framework — sources plug in via the
   interface only

**Files to create (DDL):**
```sql
CREATE TABLE ingestion_audit (
  id            TEXT PRIMARY KEY,
  source_id     TEXT NOT NULL,
  fetched_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  url           TEXT NOT NULL,
  http_status   INT,
  payload_hash  TEXT,
  payload_bytes INT,
  error         TEXT,
  duration_ms   INT
);
CREATE INDEX idx_ingestion_audit_source ON ingestion_audit (source_id, fetched_at DESC);
```

**Verification:**
```bash
# After implementing, run a no-op fetch via the framework
npx ts-node server/ingestion/test/smokeTest.ts
psql -c "SELECT count(*) FROM ingestion_audit;"
# expect: ≥ 1
```

**Required output:** at least 1 row, no errors.

**Dependencies:** none

## I002 — CDC respiratory surveillance ingestion (FluView / RSV / COVID)

**Status:** ⏳ Not started

**Why this matters:** Real-time regional respiratory activity directly
affects triage threshold for cough, fever, shortness of breath. When
flu activity is "high" in the patient's region, the differential
weighting for influenza increases and the disposition threshold for ED
referral tightens.

**Source:** CDC Open Data — `data.cdc.gov` (Socrata API). Free, no auth
required for basic use.

**Files to create:**
- `server/ingestion/sources/cdcRespiratory.ts` — implements
  `PublicDataSource` for CDC FluView, RSV-NET, and COVID surveillance
- `server/jobs/cdcRespiratorySync.ts` — BullMQ job, runs weekly on
  Mondays at 06:00 UTC

**Acceptance criteria:**
1. Fetches current-week FluView at state level (50 states + territories)
2. Normalizes each state's "activity level" (Low / Moderate / High /
   Very High) into a `MemoryEntry` with key pattern
   `surveillance:respiratory:<state>:<week>`
3. Scope: `global`, `verifiedBy: "external_guideline"`,
   `source: "CDC FluView <week-ending-date>"`
4. Job is idempotent (re-running same week produces no duplicate
   active entries — conflict resolution handles this)
5. After sync, the differential agent's prompt for respiratory
   complaints in a state with "High" activity should surface the
   surveillance entry under `## LEARNED PREFERENCES & PROTOCOLS`

**Verification:**
```bash
# Trigger sync manually
npx ts-node server/jobs/cdcRespiratorySync.ts --week=current

# Confirm rows
psql -c "SELECT key, content FROM clinical_memory
WHERE key LIKE 'surveillance:respiratory:%'
ORDER BY created_at DESC LIMIT 10;"
# expect: ≥ 10 rows (one per major state)

# Run a respiratory encounter and confirm the signal surfaces
ENCOUNTER_ID=$(curl -s -X POST localhost:3000/api/encounter \
  -d '{"complaintId":"cough","patientInput":{"age":45,"sex":"F","state":"NM","vitals":{"hr":92,"temp_c":38.5}}}' \
  | jq -r '.sessionId')

curl -s localhost:3000/api/context/$ENCOUNTER_ID/prompts/differential \
  | jq '.userPrompt' | grep -c "surveillance:respiratory"
# expect: ≥ 1
```

**Required output:** all three checks pass.

**Dependencies:** I001

## I003 — openFDA drug recall and safety ingestion

**Status:** ⏳ Not started

**Why this matters:** When a patient's medication list includes a drug
that's been recalled or has a new safety alert, the medication-review
step of the pipeline needs to flag it. Currently invisible to Auralyn.

**Source:** `api.fda.gov`. Free, no key for light use, optional key for
higher rate limits.

**Files to create:**
- `server/ingestion/sources/openFdaRecalls.ts`
- `server/ingestion/sources/openFdaSafetyAlerts.ts`
- `server/jobs/openFdaSafetySync.ts` — daily at 04:00 UTC

**Acceptance criteria:**
1. Fetches drug recalls from `/drug/enforcement.json` for the last 30
   days
2. Fetches safety alerts (FDA Drug Safety Communications)
3. Normalizes each into a `MemoryEntry` with key pattern
   `safety:drug_recall:<rxcui>:<recall_id>` or
   `safety:drug_alert:<alert_id>`
4. Scope: `global`, verified
5. Memory read path in differential and disposition agents includes
   active safety entries that match any drug in the patient's
   `current_medications` immutables

**Verification:**
```bash
npx ts-node server/jobs/openFdaSafetySync.ts

psql -c "SELECT count(*) FROM clinical_memory
WHERE key LIKE 'safety:drug_%' AND status='active';"
# expect: ≥ 5 (depends on recent FDA activity)

# Test with a patient on a commonly-recalled drug
ENCOUNTER_ID=$(curl -s -X POST localhost:3000/api/encounter \
  -d '{"complaintId":"chest_pain","patientInput":{"age":58,"sex":"M","medications":["losartan 50mg qd"]}}' \
  | jq -r '.sessionId')

# Confirm safety entries for losartan (which has had recalls) appear
# in the differential prompt
curl -s localhost:3000/api/context/$ENCOUNTER_ID/prompts/differential \
  | jq '.userPrompt' | grep -i "losartan\|recall\|safety"
```

**Required output:** safety entries appear when the patient's meds
match recall RxCUIs.

**Dependencies:** I001

## I004 — DailyMed structured drug labeling

**Status:** ⏳ Not started

**Why this matters:** When prescribing or evaluating an existing
medication, the disposition agent should ground recommendations in the
FDA-approved labeling — not in the model's training data. DailyMed
provides structured indications, contraindications, dosing, and drug
interactions.

**Source:** `dailymed.nlm.nih.gov/dailymed/services`. Free, RESTful API
with no auth.

**Files to create:**
- `server/ingestion/sources/dailyMed.ts`
- `server/jobs/dailyMedSync.ts` — runs on demand (not scheduled —
  triggered when a new drug appears in `kb_master_rules` or in a
  recent encounter's medications)

**Acceptance criteria:**
1. Given an RxCUI or generic drug name, fetch the labeling SPL
2. Extract: indications, contraindications, warnings, dosage,
   interactions
3. Store as `MemoryEntry` with key `labeling:drug:<rxcui>`, scope
   global, verified
4. Cached locally — once ingested, no re-fetch unless the labeling SPL
   version changes (track via SPL set_id and version)

**Verification:**
```bash
# Ingest a known drug
curl -s -X POST localhost:3000/api/ingestion/dailymed \
  -d '{"rxcui":"617314"}' | jq

psql -c "SELECT key, source FROM clinical_memory
WHERE key='labeling:drug:617314';"
# expect: 1 row, source contains 'DailyMed SPL'
```

**Required output:** row exists with correct key.

**Dependencies:** I001

## I005 — USPSTF preventive recommendations

**Status:** ⏳ Not started

**Why this matters:** When a patient's age, sex, and risk factors meet
USPSTF criteria for a preventive screening (e.g., colorectal cancer
screening starting at 45), the disposition agent should mention it.
This is low-priority for urgent care but high-value for the
preventive-care PCP follow-up disposition path.

**Source:** USPSTF RSS feed at
`uspreventiveservicestaskforce.org/uspstf/rss-feeds/recommendations`.
Free, no auth.

**Files to create:**
- `server/ingestion/sources/uspstf.ts`
- `server/jobs/uspstfSync.ts` — weekly on Wednesdays at 05:00 UTC

**Acceptance criteria:**
1. Pulls current A/B/C/D/I-rated recommendations
2. Normalizes population criteria (age range, sex, risk factors) into
   structured fields
3. Stores as `MemoryEntry` with key `preventive:uspstf:<topic_id>`,
   scope global, verified
4. On encounter assembly, if the patient's demographics match any
   active USPSTF A/B-rated recommendation, the entry surfaces in the
   disposition prompt

**Verification:**
```bash
npx ts-node server/jobs/uspstfSync.ts

psql -c "SELECT count(*) FROM clinical_memory
WHERE key LIKE 'preventive:uspstf:%' AND status='active';"
# expect: ≥ 50 (USPSTF maintains ~80 active recommendations)
```

**Required output:** at least 50 active rows.

**Dependencies:** I001

## I006 — Pipeline consumption: surface ingested signals in agent prompts

**Status:** ⏳ Not started

**Why this matters:** I001–I005 produce data but the data is useless if
agents don't consult it. This task wires the existing memory read path
(`fetchLearnedContext` in `memoryWriters.ts`) to include the new key
patterns so they appear in prompts when relevant.

**Files to modify:**
- `server/context/memoryWriters.ts` — extend `fetchLearnedContext` to
  include `surveillance:`, `safety:`, `labeling:`, `preventive:` key
  prefixes
- `server/context/ClinicalContextManager.ts` — verify the
  `## LEARNED PREFERENCES & PROTOCOLS` section now includes these entries
  when relevant

**Acceptance criteria:**
1. `fetchLearnedContext(scope, complaint, patientMedications, patientDemographics)`
   accepts the new parameters
2. Returns surveillance entries matching the patient's state (if
   stored in patient input)
3. Returns safety entries matching any drug in
   `patientMedications`
4. Returns labeling entries for each drug in `patientMedications`
5. Returns preventive entries matching the patient's age/sex
6. Each entry appears in the appropriate agent's prompt (surveillance
   → differential; safety → disposition; labeling → disposition;
   preventive → disposition with PCP follow-up disposition)

**Verification:**
```bash
# After I002–I005 have run at least once each:
ENCOUNTER_ID=$(curl -s -X POST localhost:3000/api/encounter \
  -d '{
    "complaintId":"chest_pain",
    "patientInput":{
      "age":58,
      "sex":"M",
      "state":"NM",
      "medications":["losartan 50mg qd","atorvastatin 40mg qd"]
    }
  }' | jq -r '.sessionId')

# Differential prompt should include relevant surveillance entries
curl -s localhost:3000/api/context/$ENCOUNTER_ID/prompts/differential \
  | jq '.userPrompt' | grep -c "LEARNED PREFERENCES"
# expect: 1

# Disposition prompt should include safety/labeling/preventive entries
curl -s localhost:3000/api/context/$ENCOUNTER_ID/prompts/disposition \
  | jq -r '.userPrompt' | grep -cE "safety:|labeling:|preventive:"
# expect: ≥ 1
```

**Required output:** both grep counts pass.

**Dependencies:** I001, I002, I003, I004, I005
