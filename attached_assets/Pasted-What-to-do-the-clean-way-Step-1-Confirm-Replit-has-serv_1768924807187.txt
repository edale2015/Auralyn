Step 2: Ensure server/storage.ts uses Admin SDK (this is the fix)

Open server/storage.ts and look at the imports.

If you see something like:

import { initializeApp } from "firebase/app"

import { getFirestore } from "firebase/firestore"
that’s the client SDK and will trigger rules errors.

You want instead:

import admin from "firebase-admin"

admin.firestore() (Admin SDK)

Here is the exact minimal Admin SDK init you should have (usually in server/storage.ts or a server/firebaseAdmin.ts):

import admin from "firebase-admin";

if (!admin.apps.length) {
  admin.initializeApp({
    credential: admin.credential.applicationDefault(),
    projectId: process.env.FIREBASE_PROJECT_ID,
  });
}

export const db = admin.firestore();


Then inside your storage functions (getPatientByPhone, createEncounter, updateEncounter, etc.) use db.collection("...")....

✅ If you do this, Firestore rules are irrelevant for server writes.

Step 3: IAM roles (what permissions matter for the service account)

In Google Cloud IAM for medicalm-dec9d, ensure the service account has one of these roles:

Cloud Datastore User (Firestore uses Datastore permissions under the hood)
or

Firestore Admin (broad, fine for dev)

This is a Google Cloud role, not a Firestore Rules thing.

Temporary Firestore Rules “allow read, write: if true”
