Something shifted this year that we need to talk about.

RAG used to be a separate pipeline. We built a retriever, bolted it onto an LLM, and called it a day. That architecture is dead now.

The agent is the retrieval system.. it decides when to search, which strategy to use, grades its own results, and loops back if the answer is not grounded.

Hit the paywall? No worries, read the full post here for free.

Some recent research and papers formalized this into three principles: autonomous strategy, iterative execution, and interleaved tool use.

If our RAG system does not satisfy all three, we are building yesterday’s architecture.

Let me walk through the five retrieval architectures that production teams are actually shipping right now, plus the infrastructure that makes them hold up.. with working code for each.

Hybrid retrieval is no longer optional
Pure vector search misses exact matches, pure BM25 misses semantics.

Production RAG today runs both in parallel, merges results with Reciprocal Rank Fusion, and optionally reranks with a cross-encoder.

The numbers are hard to argue with..

Weaviate makes this a one-liner:

results = client.query.get(
    "Document", ["content"]
).with_hybrid(
    query="contract termination procedures",
    alpha=0.7  # 0 = pure BM25, 1 = pure vector
).with_limit(10).do()
The alpha parameter controls the blend. For legal, compliance, and medical domains where exact terminology matters, we push alpha toward 0.5. For conversational queries, closer to 0.8.

Redis, Qdrant, and Pinecone all support hybrid natively now.

If our vector database does not, it is time to switch.

Self-correcting RAG with LangGraph
The most valuable upgrade most teams should adopt first.

Instead of blindly trusting retrieval results, an LLM grades them for relevance before generating an answer. If the documents are weak, it rewrites the query and falls back to web search.

This is Corrective RAG (CRAG), implemented as a state machine in LangGraph:

from langgraph.graph import StateGraph, START, END

# Each node is a function: retrieve(), grade_documents(),
# generate(), transform_query(), web_search()
# Full definitions in the CRAG tutorial linked below
workflow = StateGraph(GraphState)
workflow.add_node("retrieve", retrieve)
workflow.add_node("grade_documents", grade_documents)
workflow.add_node("generate", generate)
workflow.add_node("rewrite_query", transform_query)
workflow.add_node("web_search", web_search)
workflow.add_edge(START, "retrieve")
workflow.add_edge("retrieve", "grade_documents")
workflow.add_conditional_edges(
    "grade_documents", decide_to_generate,
    {"rewrite_query": "rewrite_query", "generate": "generate"}
)
workflow.add_edge("rewrite_query", "web_search")
workflow.add_edge("web_search", "generate")
workflow.add_edge("generate", END)
app = workflow.compile()
Add a hallucination grader after generation and we get the full adaptive stack, the system checks both retrieval quality and answer quality before responding.

In practice, this catches fabricated answers that would otherwise reach users.

LangGraph has replaced simple chains as the default orchestration pattern for one reason, agents need loops, not pipelines.

DSPy: programming agents, not prompting them
This is the one most developers are sleeping on.

DSPy from Stanford NLP treats prompts as learnable parameters instead of hand-crafted strings. We define what we want (a Signature), choose how (a Module like ChainOfThought or ReAct), and DSPy’s optimizer figures out the best prompts automatically.

A complete RAG system in DSPy:

import dspy

class RAG(dspy.Module):
    def __init__(self):
        self.retrieve = dspy.Retrieve(k=5)
        self.respond = dspy.ChainOfThought(
            "context, question -> answer"
        )
    def forward(self, question):
        context = self.retrieve(question).passages
        return self.respond(
            context=context, question=question
        )
# Auto-optimize prompts on training data
optimizer = dspy.MIPROv2(metric=SemanticF1, auto="light")
optimized_rag = optimizer(RAG(), trainset=train_examples)
That last line is the magic.
MIPROv2 tests prompt variations, finds the best examples for in-context learning, and compiles the whole thing.

Teams report going from 42% to 61% SemanticF1 with zero manual prompt tuning.

DSPy also supports dspy.ReAct for building agents with tools, and the optimizer works on those too.

Databricks has native DSPy integration for production deployment.

LightRAG: graph retrieval that actually runs
Microsoft’s GraphRAG proved that knowledge graphs dramatically improve answers for complex, multi-hop questions. But it requires expensive LLM-powered indexing and heavy infrastructure.

LightRAG from HKU solves this. Published at EMNLP, it builds a lightweight knowledge graph with dual-level retrieval, low-level for specific entity lookups, high-level for thematic understanding.

Runs on a laptop. No Neo4j required.

from lightrag import LightRAG, QueryParam

rag = LightRAG(working_dir="./rag_index")
await rag.ainsert(documents)
# Four retrieval modes: naive, local, global, hybrid
result = await rag.aquery(
    "What are the main investment themes?",
    param=QueryParam(mode="hybrid")
)
The hybrid mode combines entity-level precision with corpus-level synthesis.

For questions like "What are the main themes in this dataset?", the exact kind of question where vanilla vector RAG falls apart, LightRAG consistently outperforms both naive RAG and full GraphRAG on benchmarks.

RAG-Anything extends LightRAG to handle multimodal documents (images, tables, equations) in one pipeline.

MCP: the universal agent-tool layer
The Model Context Protocol, created by Anthropic and now governed by the Linux Foundation, is becoming the USB-C of AI agents.

Instead of writing custom API glue for every tool, MCP provides one standardized interface for agents to discover and use tools, data sources and external systems.

Every major lab adopted it.

For RAG specifically, MCP changes the game because retrieval becomes just another tool the agent can invoke strategically, rather than a fixed step that runs on every query:

from agents import Agent, Runner, FileSearchTool, WebSearchTool

agent = Agent(
    name="ResearchAgent",
    instructions="Search the knowledge base when you need "
                 "factual grounding. Use web search for "
                 "current events. Skip retrieval for "
                 "questions you can answer directly.",
    tools=[
        FileSearchTool(
            max_num_results=3,
            vector_store_ids=["vs_company_docs"]
        ),
        WebSearchTool(),
    ]
)
result = await Runner.run(agent, "What changed in our Q4 policy?")
print(result.final_output)
The agent decides whether to retrieve at all.

This is the core of adaptive RAG, the model participates in retrieval decisions instead of retrieval happening to it.

How these compose together
The strongest production systems I have seen this year are not using one of these patterns. They are layering them.

Hybrid retrieval as the search backbone.

LangGraph or DSPy for orchestration and self-correction loops.

LightRAG for domains where entity relationships matter.

MCP as the standardized interface connecting everything.

Semantic caching cuts the bill
Before any of these hits the LLM, add a semantic cache. Identical and near-identical queries get served from cache instead of burning tokens.

Redis makes this straightforward with RedisVL:

from redisvl.extensions.llmcache import SemanticCache

llmcache = SemanticCache(
    name="rag_cache",
    redis_url="redis://localhost:6379",
    distance_threshold=0.1  # tighter = fewer false hits
)
# Check before calling the LLM
if cached := llmcache.check(prompt=user_query):
    return cached[0]["response"]
# On cache miss, generate and store
response = generate_answer(user_query)
llmcache.store(prompt=user_query, response=response)
Measure it or it does not count
None of these matters if we cannot prove the system is actually answering correctly.

RAGAS gives us the metrics that matter for RAG, faithfulness (is the answer grounded in the retrieved context?), relevancy (does it actually answer the question?), and context precision (did we retrieve the right documents?).

from ragas import evaluate
from datasets import Dataset

eval_data = Dataset.from_dict({
    "question": ["What is our refund policy?"],
    "answer": ["Refunds are processed within 14 days."],
    "contexts": [["Our refund policy allows returns within "
                   "14 business days of purchase."]],
    "ground_truth": ["Refunds take 14 business days."]
})
result = evaluate(eval_data)
print(result)
# {'context_precision': 0.92, 'faithfulness': 0.95,
#  'answer_relevancy': 0.88}
Wire this into CI. Run it on every retrieval pipeline change.

Teams using Langfuse or MLflow tracing alongside RAGAS are catching retrieval regressions that blind deployments miss entirely.

Where to start
Start with hybrid retrieval and CRAG. That combination alone fixes the most common production failures, irrelevant retrieval and hallucinated answers.

Add semantic caching to cut costs, add RAGAS to prove it works.

Then layer up.

DSPy when prompt tuning becomes a bottleneck.
LightRAG when questions require connecting information across documents.
MCP when the agent needs to read and write across systems.

The tools are open source. The papers are public. The stacks are production-tested.

Build.
