# Adding a New Tab to MasterRuleMapPage

`client/src/pages/MasterRuleMapPage.tsx` is already 1,740+ lines with
12 tabs. Adding a new tab follows a strict pattern to keep it
maintainable.

## The pattern

### Step 1: Build the tab content as a separate component

NEVER add tab content inline in `MasterRuleMapPage.tsx`. The page is
too big already. Make a component:

```tsx
// client/src/components/<TabName>Panel.tsx
import { useEffect, useState } from "react";

export function <TabName>Panel() {
  // ... self-contained component logic
  return (
    <div className="space-y-4">
      {/* tab content */}
    </div>
  );
}
```

### Step 2: Import and add the tab trigger

In `MasterRuleMapPage.tsx`, find the existing `<TabsList>` and add the
new trigger in the appropriate position. Tabs are ordered by user
priority — Context Health is last because it's monitoring; new
operational tabs should go near related existing ones.

```tsx
<TabsTrigger value="<tab-key>">{/* Tab Label */}</TabsTrigger>
```

### Step 3: Add the tab content panel

After the existing `<TabsContent>` blocks:

```tsx
<TabsContent value="<tab-key>" className="space-y-4">
  <<TabName>Panel />
</TabsContent>
```

### Step 4: If the tab needs backend data, add a route

For data-bearing tabs (Context Health style), create the route in
`server/routes/<feature>.routes.ts`, register at `server/index.ts`, and
fetch from the component with React Query.

```tsx
const { data, isLoading } = useQuery({
  queryKey: ["<feature>", "<viewName>"],
  queryFn: () => fetch("/api/<feature>/<endpoint>").then(r => r.json()),
});
```

## Authentication

Every tab is auth-gated by the parent page (already in place). New
backend routes for these tabs should require physician/admin role —
follow the existing auth middleware pattern in
`server/routes/contextHealth.routes.ts`.

## Styling conventions

- Tailwind utility classes (the project already uses them throughout)
- Shadcn/ui components for primitives (Card, Tabs, Badge, etc.) —
  already imported in the page
- Color coding for health indicators: green (healthy), yellow (warn),
  red (action needed) — see ContextHealthPanel for examples
- Auto-refresh toggles for monitoring tabs use the same pattern as
  ContextHealthPanel

## What to do BEFORE writing any code

Read the existing similar tab. If you're adding a monitoring tab, read
`ContextHealthPanel.tsx`. If you're adding a data-browsing tab, read
the Rule Catalog tab. Match the existing patterns — don't introduce a
new styling vocabulary or query pattern.

## What NOT to do

- **Do not add tab content inline** in `MasterRuleMapPage.tsx`. Make a
  separate component file.
- **Do not duplicate state between tabs.** If two tabs need the same
  data, lift it via React Query — same `queryKey` shares the cache.
- **Do not bypass auth.** Even an admin-only "internal" tab needs the
  auth middleware. Internal users still need RLS context set on their
  queries.
- **Do not introduce a new UI library.** The project uses shadcn/ui +
  Tailwind. Adding a third option fragments the UI.
- **Do not add a tab without a route registered** if the tab fetches
  data. Empty tabs are confusing.

## Verification

After adding:

```bash
# Compile clean
npx tsc --noEmit

# Lint
npm run lint

# Visual: open /rule-map (authenticated) and confirm:
# - Tab appears in trigger row
# - Clicking it renders the panel
# - Data loads (or shows a clear loading state)
# - Errors are handled gracefully
```

## When the tab is part of the 510(k) story

If the tab surfaces clinical decision support, audit, or compliance
data:
- Document it in the user-facing UI documentation for the submission
- Confirm the data shown respects RLS (a physician should not see
  another physician's RLHF deltas surfaced in a UI just because they're
  in the same tenant — unless that's intentional)
- Confirm the tab does not allow editing of regulated content without
  appropriate audit trail
