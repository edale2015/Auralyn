You now have:

What you don’t yet have is feedback about how the system performs in the wild.

That’s the next layer.

6A — Add a Case Analytics Log

Add a new CSV:

server/data/csv/CASE_ANALYTICS_LOG.csv

Headers:

TIMESTAMP,CASE_ID,CC_ID,DISPOSITION,TOP_DX,DX_SCORE,RED_FLAG_TRIGGERED,PROFILE_USED,ENGINE_VERSION

Write to it inside runGenericComplaintV1() after disposition is determined.

Example row:

2026-03-05T21:41:03Z,case_9183,cough,HOME_CARE,viral_uri,0.84,false,PULM_COUGH_V3,GENERIC_V1

Purpose:

identify frequent complaints

identify diagnoses that dominate clusters

detect rules that never fire

6B — Build a Coverage Heatmap Script

Create:

scripts/cluster-coverage-report.ts

It should calculate:

CC_ID
CSR_ROWS
CLUSTERS
ACTIVE_RULES
FIRED_IN_TESTS
FIRED_IN_ANALYTICS

Output:

data/complaints/cluster_heatmap.csv

This lets you see things like:

appendicitis cluster never triggered
derm_cellulitis rules dominate rash
6C — Add “confidence calibration”

Right now your candidate score is:

score = points × clusterPriority

Add a confidence normalization layer:

confidence = score / sum(top 5 scores)

Then you can display:

Likely causes:
• Viral URI (63%)
• Pneumonia (21%)
• Asthma flare (11%)

This dramatically improves user trust.
