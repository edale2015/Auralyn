C) Encounter linking: add encounter.intakeCaseId and embed export pack in CaseDetail

This is the “workflow unification” step.

C1) Data model change

Wherever encounters are stored (Firestore/SQLite/etc), add:

intakeCaseId?: string

You don’t need migrations immediately if schema is flexible.

C2) Add server endpoint to link an intake case to an encounter

Add provider-only endpoint:

POST /api/provider/encounter/:encounterId/link-intake

Body:

{ "intakeCaseId": "CASE_..." }


Auth:

X-Provider-Key

Server behavior:

update encounter record:

intakeCaseId = body.intakeCaseId

optionally also update the case:

external_ehr.encounterId = encounterId

external_ehr.syncStatus = "ready"

Return:

{ "ok": true }

C3) Add UI control to link

Best place: CaseDetail (since that’s where the provider lives).

Add a small “Intake Case” card:

If encounter already has intakeCaseId:

show badge “Linked”

show the Export Pack panel using that case id

If not linked:

input field: paste intakeCaseId (or select from a dropdown later)

button: “Link Intake Case”

UX option (easy, very useful)

In /provider/case, add a “Copy Case ID” button next to each case so linking is easy.

C4) Render EhrExportPack in CaseDetail

In CaseDetail.tsx:

{encounter.intakeCaseId ? (
  <EhrExportPack caseId={encounter.intakeCaseId} />
) : (
  <LinkIntakeCaseCard encounterId={encounter.id} />
)}


Where LinkIntakeCaseCard calls your new link endpoint.

✅ End result: providers don’t have to bounce between “encounters” and “intake cases.” It’s one screen.
