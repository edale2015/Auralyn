Introduction — The New Age of AI Automation
A few years ago, “using AI” meant asking ChatGPT a question and copying the answer. That era is over. Today, AI can browse the web, write and run code, read your emails, update your database, call APIs, make decisions, loop until a task is done, and hand work off to another AI agent when it’s finished. It can work for hours without you touching a keyboard.

But this new power comes with new complexity. To build systems like this, you need tools — and there are a lot of them. The six tools in this guide form what many engineers and builders now call the modern AI automation stack:

n8n — connects everything with a visual workflow builder
LangChain — gives AI the ability to reason, retrieve, and use tools
LangSmith — shows you what your AI is actually doing, step by step
LangGraph — builds AI agents that can loop, branch, and persist state
CrewAI — creates teams of AI agents that collaborate on complex tasks
AutoGen — lets multiple AI agents converse and solve problems together
Some of these overlap. All of them are genuinely useful. By the end of this guide you’ll know exactly what each one does, when to reach for it, and how to start using it — without wasting weeks on tutorials that assume you already know everything. Let’s go.

1. n8n — Automate Everything Without Code
What Is n8n?

n8n (pronounced “n-eight-n”) is a workflow automation tool. It lets you connect apps, services, and APIs together visually — by drawing lines between boxes on a canvas — so that when something happens in one place, something else happens automatically somewhere else.

Think of it like this: IFTTT and Zapier had a baby, and that baby went to engineering school. It’s much more powerful than Zapier, but still visual and approachable. Unlike Zapier, n8n is open source — you can run it on your own computer or server for free, keeping your data private. With n8n you can automate things like:

“When a new row is added to Google Sheets, send a Slack message and create a Notion page”
“Every morning at 8am, fetch the top 5 news headlines and email them to me”
“When a customer submits a form, save it to a database, send a confirmation email, and create a CRM entry”
“When a GitHub issue is labeled ‘bug’, create a Jira ticket and notify the team in Discord”
But n8n goes further than most automation tools: it has built-in AI nodes. You can add an AI agent into any workflow, give it tools, and let it make decisions mid-flow. This is where n8n crosses from “automation tool” into “AI automation platform.”

Who Uses n8n?

Developers who want automation without building an entire backend
Non-technical teams who need powerful automation without code
Startups automating repetitive processes (lead routing, customer onboarding, reporting)
Data teams moving data between systems
AI builders connecting AI models to real business workflows
Core Concepts (Plain English)
Workflow: A series of connected steps that run automatically. Like a recipe — do Step 1, then Step 2, then Step 3. One workflow might have 3 steps; another might have 30.

Node: One step in your workflow. Each node is a box on the canvas. There are nodes for hundreds of apps and services: Google Sheets, Slack, Gmail, PostgreSQL, HTTP requests, Stripe, Airtable, Notion, and hundreds more. There are also nodes for code (write custom JavaScript or Python), logic (if/else, loops, merge), and AI (ChatGPT, Claude, Gemini).

Trigger Node: The first node in every workflow. It defines what starts the workflow. Types of triggers:

Webhook — starts when an external service sends a request to n8n’s URL
Schedule — starts at a set time (every hour, every Monday at 9am)
App trigger — starts when something happens in a connected app (new Gmail email, new Stripe payment)
Manual — you press a button to run it
Connection: The arrow between two nodes. Data flows along connections from left to right. The output of one node becomes the input of the next.

Item: n8n processes data as “items” — one item is one unit of data (one row, one email, one order). If your workflow receives 10 emails, it processes 10 items, one at a time (or in batches).

Expression: A small piece of code you write inside a node’s field to make it dynamic. Example: instead of typing a fixed email subject, you write {{ $json.customerName }}'s order is ready — and n8n fills in the actual customer name from the data flowing through. Expressions use {{ }} syntax and can access any data from any previous node.

AI Agent Node: n8n’s most powerful node. It connects to an LLM (like GPT-4 or Claude), gives it tools (like web search or database queries), and lets it work autonomously until it completes a task. The AI decides which tools to use and in what order — you just define the available tools and the goal.

Memory Node: Works with the AI Agent node to give the AI memory across conversations or workflow runs. Without this, the AI starts fresh every time.

Credential: A stored connection to an external service — your API key, username/password, or OAuth token. You set up credentials once and reuse them across all nodes of that type. Credentials are encrypted and stored securely.

Sub-workflow: A workflow that gets called from inside another workflow. Lets you build reusable chunks of automation — like functions in programming.

Step 1 — Create Your First Workflow

Click “New Workflow.” You see an empty canvas. In the center is a “+” button — click it to add your first node.

Step 2 — Add a Trigger

Search for “Schedule Trigger.” Set it to run every day at 8am. This workflow will run automatically on schedule.

Step 3 — Add an HTTP Request Node

Click the “+” on the right side of the Schedule node. Search for “HTTP Request.” This node can call any API. Set:

Method: GET
URL: https://hacker-news.firebaseio.com/v0/topstories.json
This fetches the top story IDs from Hacker News.

Step 4 — Add a Code Node

Add a Code node after HTTP Request. Write:

const ids = $input.first().json.slice(0, 5);
return ids.map(id => ({ json: { id } }));
This takes the first 5 story IDs and creates 5 items.

Step 5 — Add Another HTTP Request

This one fetches the details of each story:

URL: https://hacker-news.firebaseio.com/v0/item/{{ $json.id }}.json
The {{ $json.id }} is an expression — n8n fills in each story's ID automatically.

Step 6 — Add a Gmail Node

Add “Gmail → Send Email.” Set:

To: your email address
Subject: Top Hacker News Stories
Body: {{ $json.title }} - {{ $json.url }}
Step 7 — Test and Activate

Click “Test Workflow” to run it manually. Check your email. If it works, click “Active” (top right toggle). The workflow now runs automatically every morning.

Intermediate Skills
Using the IF Node (Conditional Logic): Add an IF node to branch your workflow. “If the order total is over $500, send to the high-value customer flow. Otherwise, send to the standard flow.” Set conditions using expressions: {{ $json.orderTotal }} > 500.

Looping with the Loop Over Items Node: If you have 100 items to process but want to handle them 10 at a time (to avoid rate limits), use the Loop node. Set batch size to 10 and n8n automatically chunks and loops.

Error Handling: Right-click any node → “Add Error Output.” Connect the error output to a Slack node that notifies you when something fails. Never let a broken workflow silently fail.

Using the Merge Node: When two branches of a workflow need to come back together before continuing, the Merge node combines their outputs. Three modes: Append (stack all items), Merge by Position (line them up 1-to-1), Merge by Key (match items by a shared field like id).

Passing Data Between Nodes: Every node can see all data from all previous nodes using $('Node Name').all() syntax. Not just the immediately previous node — any node by name. This is powerful when you need data from step 2 while you're in step 7.

Webhooks for Real-Time Triggers: Add a Webhook node as your trigger. n8n gives you a unique URL. Any service that supports webhooks (GitHub, Stripe, Typeform, Shopify) can hit that URL and start your workflow immediately, with the payload data available in all downstream nodes.

Connecting to LangChain: n8n has a “LangChain” section of nodes that wrap LangChain concepts natively:

Environment Variables and .env Files: For self-hosted n8n, sensitive configuration (API keys, database URLs) should live in environment variables, not hardcoded in workflows:

export DB_TYPE=postgresdb
export DB_POSTGRESDB_HOST=localhost
Scaling n8n with Queues: For production use with high volume, enable Queue Mode. This uses Redis as a message queue and lets you run multiple n8n worker processes in parallel:

N8N_MODE=queue
QUEUE_BULL_REDIS_HOST=localhost
Common Mistakes and How to Avoid Them
Mistake 1: Using Set nodes to pass data instead of expressions. Beginners add a Set node after every step to “save” a value. You don’t need to — use expressions {{ $('Step Name').first().json.field }} to access any previous data directly.

Mistake 2: Not handling errors. A workflow that fails silently is worse than one that never ran. Add error outputs to critical nodes and connect them to a notification.

Mistake 3: Storing credentials in node fields. Never paste API keys directly into node fields. Use n8n’s Credentials system — it encrypts them and lets you reuse them across nodes.

Mistake 4: Running the whole workflow to test one node. Right-click any node → “Execute Node” to run just that one step with test data. You don’t need to trigger the whole workflow every time.

Mistake 5: Ignoring the execution log. n8n saves a log of every run — go to “Executions” in the left sidebar to see what happened, what data flowed through each node, and where it failed.

2. LangChain — The AI Plumbing Kit
What Is LangChain?

LangChain is an open-source Python and JavaScript library that helps developers build applications powered by large language models (LLMs). It doesn’t provide the AI — it provides the connective tissue between your app, your data, and the AI.

Without LangChain: connecting an AI to your documents requires hundreds of lines of custom code. With LangChain: you snap together pre-built components in a few dozen lines.

LangChain is the most widely used AI framework in the world. Its ecosystem includes:

LangChain Core — the base library (prompts, models, chains)
LangChain Community — integrations with 300+ services (vector stores, tools, loaders)
LangGraph — advanced agent framework (covered separately below)
LangSmith — observability and debugging (covered separately below)
Core Concepts (Plain English)
LLM / Chat Model: The brain. LangChain wraps AI models from OpenAI, Anthropic, Google, Mistral, Cohere, Ollama (local models), and dozens more. Switching providers is changing one line of code.

from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic

# OpenAI
model = ChatOpenAI(model="gpt-4o")

# Anthropic Claude
model = ChatAnthropic(model="claude-3-5-sonnet-20241022")
Messages: LangChain uses a message format for all conversations:

SystemMessage — instructions to the AI ("You are a helpful assistant")
HumanMessage — the user's input
AIMessage — the AI's response
from langchain_core.messages import SystemMessage, HumanMessage

messages = [
    SystemMessage(content="You are a pirate. Always speak in pirate style."),
    HumanMessage(content="What is the capital of France?")
]

response = model.invoke(messages)
print(response.content)  # "Arr, 'tis Paris, matey!"
Prompt Template: A reusable skeleton for prompts with placeholders.

from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are an expert in {subject}. Answer concisely."),
    ("human", "{question}")
])

chain = prompt | model
result = chain.invoke({"subject": "astronomy", "question": "Why is Pluto not a planet?"})
LCEL (LangChain Expression Language): The | pipe operator. Chain components together:

chain = prompt | model | output_parser
result = chain.invoke({"key": "value"})
Left to right: prompt formats the text → model generates a response → parser structures the output. LCEL also enables streaming, batching, and parallel execution out of the box.

Output Parser: Structures the AI’s response into a usable format.

from langchain_core.output_parsers import StrOutputParser, JsonOutputParser
from pydantic import BaseModel

# Simple string output
chain = prompt | model | StrOutputParser()

# Structured JSON output
class Product(BaseModel):
    name: str
    price: float
    category: str

parser = JsonOutputParser(pydantic_object=Product)
Document Loader: Reads content from different sources and standardizes it.

from langchain_community.document_loaders import (
    PyPDFLoader,          # PDFs
    WebBaseLoader,        # Websites
    CSVLoader,            # CSV files
    NotionDBLoader,       # Notion databases
    YoutubeLoader,        # YouTube transcripts
    GitLoader,            # GitHub repos

)

loader = PyPDFLoader("company-handbook.pdf")

Text Splitter: Breaks large documents into smaller chunks the AI can process.

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,      # Each chunk is ~500 characters
    chunk_overlap=50,    # 50 characters overlap between chunks (preserves context at boundaries)
    separators=["\n\n", "\n", ". ", " "]  # Split preferring paragraph breaks first
)

chunks = splitter.split_documents(docs)
Embeddings: Converts text into a list of numbers that capture meaning. Similar text → similar numbers.

from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vector = embeddings.embed_query("What is the refund policy?")
# Returns: [0.023, -0.145, 0.891, ...] — 1536 numbers
Vector Store: A database that stores embeddings and finds similar ones fast.

from langchain_community.vectorstores import Chroma, FAISS
from langchain_pinecone import PineconeVectorStore

# Local (development)
vectorstore = FAISS.from_documents(chunks, embeddings)

# Production (managed)
vectorstore = PineconeVectorStore.from_documents(
    chunks, embeddings, index_name="my-index"
)

# Search
results = vectorstore.similarity_search("refund policy", k=3)
Retriever: The interface that fetches relevant documents from a vector store.

retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 4}  # Return top 4 most similar chunks
)

docs = retriever.invoke("What is the cancellation policy?")
RAG (Retrieval Augmented Generation): The most important LangChain pattern. AI answers from your data, not its training.

from langchain_core.runnables import RunnablePassthrough

rag_chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | ChatPromptTemplate.from_template(
        "Answer based on this context:\n{context}\n\nQuestion: {question}"
    )
    | model
    | StrOutputParser()
)

answer = rag_chain.invoke("What is the refund policy?")
Memory: Gives AI conversation history.

from langchain_community.chat_message_histories import ChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory

store = {}

def get_session_history(session_id):
    if session_id not in store:
        store[session_id] = ChatMessageHistory()
    return store[session_id]

chain_with_memory = RunnableWithMessageHistory(chain, get_session_history)

# Session "user123" will remember previous messages
chain_with_memory.invoke(
    {"question": "My name is Alex"},
    config={"configurable": {"session_id": "user123"}}
)
chain_with_memory.invoke(
    {"question": "What's my name?"},
    config={"configurable": {"session_id": "user123"}}
)
# Output: "Your name is Alex."
Tools: Functions the AI can call to interact with the world.

from langchain.tools import tool

@tool
def get_stock_price(ticker: str) -> str:
    """Get the current stock price for a given ticker symbol like AAPL or TSLA."""
    # call a stock API here
    return f"{ticker}: $182.50"

@tool
def send_email(to: str, subject: str, body: str) -> str:
    """Send an email to a recipient."""
    # call email API here
    return f"Email sent to {to}"

Full RAG pipeline:

from langchain_community.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

# Load and process documents
loader = PyPDFLoader("your-document.pdf")
docs = loader.load()
chunks = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50).split_documents(docs)

# Store as embeddings
vectorstore = FAISS.from_documents(chunks, OpenAIEmbeddings())
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# Build the RAG chain
prompt = ChatPromptTemplate.from_template("""
Answer the question based only on this context:
{context}

Question: {question}
""")

rag_chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | ChatOpenAI(model="gpt-4o-mini")
    | StrOutputParser()
)

answer = rag_chain.invoke("What are the main topics covered?")
print(answer)

Common Mistakes
Mistake 1: Using outdated APIs. LangChain deprecated many old classes (LLMChain, ConversationalRetrievalChain). Use LCEL. Always check docs at python.langchain.com.

Mistake 2: One chunk size fits all. 500-character chunks work for articles. For legal documents, try 1500. For code, split by function, not character count. Tune chunk size for your content type.

Mistake 3: Forgetting chunk overlap. Without overlap, a sentence split across two chunks loses context in both. Always set chunk_overlap to 10-20% of chunk size.

Mistake 4: Not tracking costs. Use LangSmith (next section) to see exactly how many tokens each call uses and what it costs.

Mistake 5: Using powerful models for simple tasks. GPT-4o for a yes/no classification? Use GPT-4o-mini — it’s 95% as good at 5% of the cost.

3. LangSmith — See Inside Your AI
What Is LangSmith?
LangSmith is Anthropic’s observability and debugging platform for AI applications — specifically for LangChain and LangGraph apps. It records every single thing that happens inside your AI pipeline: what prompt went in, what the AI returned, which tools were called, how long each step took, and what it cost.

Without LangSmith: building AI apps is like debugging a program with no print statements, no logs, and no error messages. Things go wrong and you have no idea why. With LangSmith: every run is recorded as a tree you can inspect. You see the full conversation, the exact prompt that was sent, the raw model response, every tool call, and every retrieval result. It turns AI debugging from guesswork into a science.

LangSmith also adds:

Datasets — store example inputs/outputs for testing
Evaluations — automatically test your AI pipeline against datasets
Monitoring — track latency, cost, error rates, and quality in production
Annotations — let humans review and label AI outputs
Playground — test prompts interactively without writing code
Who Uses LangSmith?
Developers debugging why their AI agent made a wrong decision
Teams running automated tests on AI pipelines before deploying
Engineers monitoring production AI apps for quality degradation
Researchers experimenting with different prompts and models
Core Concepts (Plain English)
Trace: One complete recording of a workflow run. When you call your LangChain chain, LangSmith records a Trace. Inside the trace, you see every step — prompts, completions, tool calls, retrieval results — as a nested tree.

Trace: "Answer customer question about refunds"
  └── ChatPromptTemplate  [2ms]
  └── ChatOpenAI         [1.2s]  — Cost: $0.003
  └── StrOutputParser    [0ms]
Span: One step inside a trace. Each LangChain component creates a span — you can see exactly how long it took, what went in, and what came out.

Run: One execution of a single component. A trace is made of many runs — one per component.

Project: A container for traces. You create separate projects for different apps or environments (development, staging, production).

Dataset: A collection of example inputs and expected outputs. Used for automated testing.

Example: 50 customer questions with the correct answers
LangSmith runs your AI pipeline against all 50
Compares AI answers to expected answers
Reports which ones passed and failed
Evaluator: A function (or another AI model) that judges whether a run’s output is correct, relevant, and high quality. Types:

Exact match — is the output exactly this string?
Semantic similarity — is the meaning close enough?
LLM-as-judge — ask GPT-4 to rate whether the answer is correct
Custom — write your own scoring function
Feedback: A score attached to a run — thumbs up/down, 1–5 stars, or custom metrics. You can add feedback programmatically or let users submit it from your app.

Monitoring Dashboard: Production metrics over time — P50/P95 latency, token usage, cost, error rate, feedback scores. Set up alerts when error rate spikes or latency degrades.

Getting Started: Step by Step
Step 1 — Create a LangSmith Account

Go to smith.langchain.com. Sign up free. Create a project called “my-first-project.”

Step 2 — Get Your API Key

Go to Settings → API Keys → Create API Key. Copy it.

Step 3 — Enable Tracing in Your Code

Add these lines at the top of your Python file (or in your .env):

import os
os.environ["LANGCHAIN_TRACING_V2"] = "true"
os.environ["LANGCHAIN_API_KEY"] = "your-langsmith-api-key"
os.environ["LANGCHAIN_PROJECT"] = "my-first-project"
Or in .env:

LANGCHAIN_TRACING_V2=true
LANGCHAIN_API_KEY=ls__...
LANGCHAIN_PROJECT=my-first-project
That’s it. No other code changes needed. Every LangChain call now automatically sends traces to LangSmith.

Step 4 — Run Your Code

Run any LangChain code. Go to smith.langchain.com → your project → you’ll see runs appearing in real time.

Step 5 — Explore a Trace

Click any run. You see a tree of every step. Click on the ChatOpenAI node and see:

Output: The exact raw response
Latency: 1.23s
Tokens: 450 input, 120 output

from langsmith.evaluation import evaluate, LangChainStringEvaluator

def predict(inputs):
    return {"answer": rag_chain.invoke(inputs["question"])}

    predict,
    data="customer-service-qa",
    evaluators=[evaluator],
    experiment_prefix="rag-v1"
)

4. LangGraph — AI That Thinks in Loops
What Is LangGraph?

LangGraph is a framework for building stateful, multi-step AI agents. It’s part of the LangChain ecosystem but is a completely separate library designed to solve a specific problem: regular LangChain chains run in a straight line — but real-world tasks rarely work that way.

Real tasks need:

Loops (“keep searching until you find a good answer”)
Branches (“if the AI is unsure, ask a human; otherwise, continue”)
Parallel work (“research three topics at the same time”)
Memory across runs (“remember what happened last Tuesday”)
Human-in-the-loop checkpoints (“pause and ask for approval before sending that email”)
LangGraph solves all of this by letting you model your AI workflow as a graph — nodes (steps) connected by edges (what happens next). Edges can be conditional (“go to node A if this, node B if that”) and the graph can loop back to earlier nodes.

It’s the tool serious AI engineers reach for when they need agents that are reliable, controllable, and production-ready.

Who Uses LangGraph?

Engineers building complex AI agents that need to loop and retry
Teams building AI workflows with human approval steps
Developers who tried simple agents and found them too unpredictable
Production AI systems that need persistent state across sessions
Core Concepts (Plain English)
State: A dictionary that every node in the graph can read and write. This is the agent’s “working memory” — what it knows at any point in time. You define the State schema upfront.

from typing import TypedDict, List
from langchain_core.messages import BaseMessage

class AgentState(TypedDict):
    messages: List[BaseMessage]      # Conversation history
    current_plan: str                # What the agent is working on
    search_results: List[str]        # What it found
    iteration_count: int             # How many loops it's done
    is_complete: bool                # Whether it's done
Node: A Python function that takes the current State, does something, and returns an updated State.

def research_node(state: AgentState) -> AgentState:
    # Read from state
    query = state["messages"][-1].content
    
    # Do something (call an API, run the AI, search the web)
    results = search_tool.run(query)
    
    # Return updated state — only the fields that changed
    return {"search_results": [results]}
Edge: The connection between nodes. Tells the graph what to do after a node finishes.

Direct edge — always go to this next node: graph.add_edge("research", "write")
Conditional edge — decide at runtime where to go next: graph.add_conditional_edges("check", decide_next_step)
Conditional Edge Function: A Python function that looks at the current State and returns the name of the next node to run.

def decide_next(state: AgentState) -> str:
    if state["is_complete"]:
        return END               # Stop the graph
    elif state["iteration_count"] > 5:
        return "give_up"         # Too many attempts, fallback
    elif not state["search_results"]:
        return "research"        # No results yet, search again
    else:
        return "write_answer"    # We have results, write the answer
START and END: Special nodes. Every graph starts at START and runs until it reaches END.

Checkpointer: Saves the State after every node runs. This enables:

Resuming — if the agent crashes, restart from the last checkpoint
Human-in-the-loop — pause at a node and wait for human input before continuing
Long-running agents — an agent that runs for hours or days, persisting state between steps
from langgraph.checkpoint.sqlite import SqliteSaver

checkpointer = SqliteSaver.from_conn_string("agent_state.db")
graph = graph_builder.compile(checkpointer=checkpointer)
Thread: Each separate “conversation” or “task” gets its own thread ID. The checkpointer saves state per thread, so many users can run the same graph simultaneously without interfering with each other.

config = {"configurable": {"thread_id": "user-123-task-456"}}
result = graph.invoke(initial_state, config=config)
Human-in-the-Loop: You can make the graph pause at a specific node and wait for a human to provide input or approve an action.

graph = graph_builder.compile(
    checkpointer=checkpointer,
    interrupt_before=["send_email"]  # Pause before this node runs
)

# Run until it pauses
result = graph.invoke(state, config)
# Graph is now paused. Human reviews.

# Human approves — resume from where it stopped
graph.invoke(None, config)  # Resumes from the checkpoint
Building a Complete LangGraph Agent
Here’s a full research agent that searches, evaluates results, and loops if needed:

from typing import TypedDict, List, Annotated
from langchain_core.messages import HumanMessage, AIMessage
from langchain_openai import ChatOpenAI
from langchain_community.tools import DuckDuckGoSearchRun
from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import ToolNode

# 1. Define State
class ResearchState(TypedDict):
    messages: Annotated[List, "add_messages"]
    research_done: bool

# 2. Set up tools
search = DuckDuckGoSearchRun()
tools = [search]
model = ChatOpenAI(model="gpt-4o").bind_tools(tools)

# 3. Define nodes
def research_agent(state: ResearchState):
    response = model.invoke(state["messages"])
    return {"messages": [response]}

def should_continue(state: ResearchState) -> str:
    last_message = state["messages"][-1]
    # If AI used a tool, run the tool
    if hasattr(last_message, "tool_calls") and last_message.tool_calls:
        return "tools"
    # Otherwise, we're done
    return END

# 4. Build the graph
builder = StateGraph(ResearchState)
builder.add_node("agent", research_agent)
builder.add_node("tools", ToolNode(tools))

builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", should_continue)
builder.add_edge("tools", "agent")  # After tool runs, go back to agent

graph = builder.compile()

# 5. Run it
result = graph.invoke({
    "messages": [HumanMessage(content="What are the latest developments in fusion energy?")],
    "research_done": False
})
The agent loops: calls the model → model decides to search → search runs → model reads results → model decides it has enough → returns final answer.

Advanced Patterns

Multi-Agent Graph: Multiple specialized agents as nodes in one graph.

builder.add_node("researcher", researcher_agent)   # Searches and gathers info
builder.add_node("analyst", analyst_agent)         # Analyzes and evaluates info
builder.add_node("writer", writer_agent)           # Writes the final output
builder.add_node("reviewer", reviewer_agent)       # Reviews and decides if good enough

builder.add_edge("researcher", "analyst")
builder.add_edge("analyst", "writer")
builder.add_conditional_edges("reviewer", {
    "revise": "writer",      # Not good enough — rewrite
    "done": END              # Good — we're done
})

Common Mistakes
Mistake 1: Using LangGraph for simple pipelines. If your workflow is linear (A → B → C, always), LCEL is simpler and faster. Use LangGraph when you genuinely need loops or conditional branching.

Mistake 2: Not defining State schema clearly. The State is everything. If it’s ambiguous or missing fields, nodes will fail in confusing ways. Define every field upfront.

Mistake 3: Forgetting to add a max iteration check. An agent that loops can loop forever. Always add a counter to your State and a condition that breaks the loop after N iterations.

Mistake 4: Not using a checkpointer in production. Without checkpointing, a crash means the agent restarts from scratch. Use SqliteSaver for development, PostgresSaver for production.

Mistake 5: Putting too much logic in edges. Edge functions should only decide where to go next — not compute results or call APIs. Keep computation in nodes, routing in edges.

5. CrewAI — Build a Team of AI Agents
What Is CrewAI?

CrewAI is a framework for building teams of AI agents. Instead of one AI doing everything, you define multiple AI “crew members” — each with their own role, expertise, and tools — and they collaborate to complete complex tasks.

Think of it like hiring a team: you have a researcher, a writer, an editor, and a fact-checker. Each one does their part, hands work to the next, and together they produce something better than any one of them could alone.

CrewAI makes this multi-agent collaboration easy to build. You define agents, define tasks, assign tasks to agents, and CrewAI orchestrates the communication and handoffs automatically.

Who Uses CrewAI?

Builders who want multi-agent systems without deep framework knowledge
Teams automating research, content creation, and data analysis pipelines
Founders building AI products where agents take on specialized roles
Developers who want a higher-level abstraction than raw LangGraph
Core Concepts (Plain English)
Agent: One AI crew member. Has a role, a goal, a backstory (context that shapes its behavior), and tools it can use.

Common Mistakes
Mistake 1: Making agents too general. An agent with goal “do everything well” performs worse than one with goal “analyze financial data and identify anomalies.” Be specific with roles and goals.

Mistake 2: Not writing backstories carefully. The backstory shapes the agent’s personality and approach. A backstory that says “you are meticulous and always double-check facts” produces more careful outputs than “you are an AI assistant.”

6. AutoGen — AI Agents That Talk to Each Other
What Is AutoGen?

AutoGen is a framework from Microsoft Research for building multi-agent AI systems where agents converse with each other to solve problems. Instead of agents handing off output in a pipeline (like CrewAI), AutoGen agents actively talk — one agent proposes, another critiques, a third verifies — until they reach a solution together.

If CrewAI is like a team where each person does their assigned job and passes it to the next, AutoGen is like a team in a meeting room, debating and discussing until they agree on the best answer. AutoGen’s real superpower is code execution. Agents can write Python code, a code executor agent runs it, reads the output, and decides if the result is correct — fixing and rerunning if not. This makes AutoGen exceptional for data analysis, math, scientific computing, and any task where the work can be verified by running code. AutoGen 0.4 (the latest version, also called AutoGen Studio) was a major rewrite with a cleaner API called AgentChat.

Who Uses AutoGen?

Researchers building complex reasoning systems
Data scientists automating analysis pipelines with AI
Engineers building coding assistants that write and verify their own code
Teams exploring autonomous multi-agent research workflows

Common Mistakes
Mistake 1: Forgetting it’s async. AutoGen 0.4 is fully async. Use async def and await or asyncio.run(). Calling agents synchronously won't work.

Mistake 2: Not setting termination conditions. Without a termination condition, agents will converse until they hit the model’s context limit or cost you a fortune. Always set MaxMessageTermination.

Mistake 5: Ignoring AutoGen Studio. AutoGen Studio is a visual UI for building and testing multi-agent systems without code. Go to autogen-studio.com and test your agent team interactively before building the Python code.

A Real-World Example
You’re building an AI-powered competitive intelligence tool for a SaaS company:

Miro (not in this guide, but worth mentioning): Map out what the system should do.

n8n: Trigger the whole workflow every Monday morning at 9am. Pull a list of competitors from Google Sheets. For each competitor, trigger the AI pipeline. At the end, email the report to the team and save it to Notion.

CrewAI: A 3-agent crew handles each competitor:

Researcher agent (with web search tools) gathers recent news, product updates, pricing changes
Analyst agent evaluates the competitive threat and opportunities
Writer agent produces a clean, structured report section
LangChain: The Researcher agent uses LangChain’s RAG to search an internal knowledge base of past research reports, so it doesn’t repeat work and builds on previous findings.

LangGraph: If the Researcher agent can’t find enough data (less than 5 sources), a LangGraph loop triggers additional search strategies — trying different search queries and sources — before passing to the Analyst.

LangSmith: Every run is traced. You can see exactly what each agent found, what prompted each decision, and catch quality issues. When a report is surprisingly wrong, you open LangSmith and see in 30 seconds which step failed.

Result: Every Monday morning, the team wakes up to a fully researched, AI-written competitive intelligence report — built by a system you designed, orchestrated, and can trust because you can see every step it took.

Conclusion
A year ago, “building with AI” meant calling the OpenAI API and printing a response. Today it means something entirely different — orchestrating teams of AI agents, giving them tools and memory, watching them research, reason, write code, loop until they’re right, and hand finished work to the next agent in the chain. These six tools are the enablers of that shift:

n8n is where the real world meets your AI — connecting apps, APIs, and databases to your workflows with minimal code and maximum reliability.

LangChain is the foundation — the standard library of AI development. If you’re building anything with AI in Python or JavaScript, you will encounter LangChain. Learn it well.

LangSmith is the silent guardian of your AI systems. You can’t improve what you can’t see. Run it from your very first line of LangChain code and you’ll never go back to flying blind.

LangGraph is where AI becomes truly powerful. When your agents need to think in loops, persist state, and handle the unexpected, LangGraph is what makes that controllable and production-ready.

CrewAI is where collaboration happens. Complex tasks that would overwhelm one AI become manageable when split among a specialized crew — each agent doing what it does best, together achieving what none could alone.

AutoGen is where AI becomes collaborative in a different sense — agents that argue, verify each other’s work, write and run code, and won’t stop until the answer is actually right.

You don’t need to master all six at once. Here’s the honest order:

Start with LangChain — understand prompts, chains, and RAG
Add LangSmith — immediately, from day one of using LangChain
Move to n8n — when you need to connect your AI to the real world
Learn LangGraph — when your agent needs to loop or persist state
Explore CrewAI — when you want multiple specialized agents working in a pipeline
Try AutoGen — when your task benefits from agents that converse and execute code
The gap between “I have an AI idea” and “I have a working AI system” has never been smaller. These tools are the reason. Start with one. The stack will follow.

All tools update frequently — always check official documentation.
