🔧 Two improvements I recommend (small, high value)
A) Add schema validation + typed parsing guardrails

Right now rules are flexible (number/boolean/text/json), which is good, but you want to prevent silent errors like:

TAMIFLU_MAX_DAYS = "two" → parse becomes null → default used quietly

Add a strict check after loading:

if a key exists but parsed value is null, log:

[EntFluRules] Invalid value for TAMIFLU_MAX_DAYS: "two" (expected number). Using default 2.

This makes sheet editing safe for future-you.

B) Add a “rules_version” row and log it on each encounter

Add in ENT_FLU_RULES:

RULES_VERSION,text,2026-01-22a,Y

Then when computing proposal:

store proposal.rules_version

This gives you auditability: when you review outcomes later, you know exactly which rules produced a given recommendation.

🧷 One operational safeguard (prevents “it works but not in prod”)
Load flows/rules at server startup (warm cache)

load ENT_FLU questions

load ENT_FLU rules

If it fails:

log loud warning once

This prevents the first patient message from being delayed by Sheets latency, and makes failures obvious immediately.
