Issue Title: Three Divergent EHR Write Implementations With Irreconcilable Failure Semantics
Severity: Critical
Category: Architectural Risk / Real Defect
Files: ehrWriter.ts, ehrExecutor.ts, ehrOrchestrator.ts
What is wrong: The system has three independent EHR write paths. ehrWriter.ts throws on failure (correct). ehrExecutor.ts catches errors internally and returns { written: false } (incorrect — callers see a resolved promise, not an exception). ehrOrchestrator.ts never makes a real call and always returns { success: true, stub: true }. There is no single canonical write path. It is impossible to determine at runtime which path was invoked.
Why it matters: Clinical chart writes are the most consequential operation in the system. Conflicting implementations mean that which path gets called determines whether a physician is notified of failure. Two of three paths can succeed silently while no write occurred.
Evidence from code:
typescript// ehrExecutor.ts — swallows failure
} catch (err: any) {
  return { written: false, system: "athena", error: err.message };
}

// ehrOrchestrator.ts — never makes a real call
return { success: true, system: action.split("_")[0], stub: true, ... };
Recommended fix: Delete ehrOrchestrator.ts entirely (it is pure stub theater). Audit every call site for submitEncounter() and redirect to ehrWriter.ts. Fix ehrExecutor.ts adapters to throw on failure. Establish one canonical write path with a clear interface.
Type: Architectural Risk + Real Defect

Issue Title: ehrExecutor.ts Adapter Functions Return Failure Objects Instead of Throwing
Severity: Critical
Category: Real Defect

The scope gate is passed. The audit log records nothing. The physician UI receives no error.
Why it matters: This directly reproduces the silent-success-on-failure bug that ehrWriter.ts's comment claims was fixed. The fix exists in one file but not the other. Whichever path is used at runtime may silently fail.

Recommended fix: Remove try/catch from adapter functions. Let errors propagate. executeWithScope should catch and re-throw with audit logging.

Why it matters: EHR automation credentials (Athena/ECW/Epic passwords or API tokens) are a HIPAA Business Associate level secret. A misconfigured RLS policy, a SQL injection, a database dump, or a compromised read replica exposes every credential in cleartext.

...

Recommended fix: Encrypt secret_json at the application layer before write using a KMS-managed key (AWS KMS, Vault). Store only ciphertext in the DB. Decrypt on read. Never log or return decrypted secrets in API responses.

Issue Title: approvalGate.ts — Blocking 15-Minute Poll Loop Holds Automation Session Open
Severity: High
Category: Architectural Risk
Files: approvalGate.ts, actionRunner.ts, automationService.ts
What is wrong: waitForApproval() polls the database every 2 seconds for up to 15 minutes. It runs synchronously within the runAutomationAction() call chain, which itself runs inside runAutomation() which holds a live Playwright browser session. If approval never comes, the browser stays open for 15 minutes consuming memory and Chrome processes. There is no external cancellation signal.
Why it matters: In a multi-tenant environment, N concurrent pending approvals = N live Chrome instances. Each Chrome instance is 200–500MB of RAM. This is a practical denial-of-service vector and a resource exhaustion risk.
Recommended fix: Replace the polling loop with a database-backed notification (PostgreSQL LISTEN/NOTIFY) or a webhook-driven state machine. The automation run should suspend (status = awaiting_approval) and resume when the approval is recorded, not hold an open session.
Type: Architectural Risk

Issue Title: policyEngine.ts — Name-Based Regex Gate Is Trivially Circumvented
Severity: High
Category: Architectural Risk / Incomplete Scaffolding
Files: policyEngine.ts, actionRunner.ts
What is wrong: The only content-level policy enforcement is action.name matching /submit|final|send/i. An action named "confirm_chart", "execute_write", or "complete_encounter" passes with requiresApproval: false. No payload inspection occurs. No action type × payload combination triggers a review. Admin URL detection is action.url?.includes("admin") — trivially bypassed by using a non-admin-looking URL that resolves to an admin endpoint.
Why it matters: Policy engines for clinical automation must be content-aware, not name-aware. The current engine provides the appearance of governance while offering near-zero real protection.
Recommended fix: Define an explicit allowlist of (templateKey, actionType) pairs that require approval, independently of action names. Inspect payload contents for clinical write indicators (diagnosis, medication, chart update). Names must not be the security primitive.
Type: Architectural Risk

Issue Title: fhirAuth.ts — Global In-Memory Token Cache Is Not Multi-Tenant Safe
Severity: High
Category: Architectural Risk
Files: fhirAuth.ts
What is wrong: tokenCache is a module-level variable. In a multi-tenant system where different clinics use different FHIR credentials, the first clinic to acquire a token poisons the cache for all other clinics. Every subsequent FHIR call from any tenant uses the first tenant's token until it expires.
Why it matters: Cross-tenant FHIR access is a HIPAA breach. Tenant A's bearer token being used to read or write Tenant B's patient data would be an unauthorized disclosure.
Evidence from code:
typescriptlet tokenCache: TokenCache | null = null; // single global, no tenant key
Recommended fix: Key the cache by clinicId or credential set hash: Map<string, TokenCache>. Pass the tenant identifier through the call chain. Alternatively, accept that each FHIR call acquires its own token (acceptable with short-lived tokens and a proper cache-aside pattern per tenant).
Type: Architectural Risk

Issue Title: visionAgent.ts — selectorMemory and uiMemory Are Process-Global, Not Tenant-Scoped
Severity: Medium
Category: Architectural Risk
Files: visionAgent.ts
What is wrong: selectorMemory and uiMemory are plain module-level objects. Selector learning from one clinic's automation run persists and can be recalled by any other clinic's run in the same process. A learned selector for Athena Clinic A can be returned to Clinic B's automation session.
Why it matters: Selector leakage across tenants is a functional defect. It will cause incorrect automation behavior in multi-tenant deployments and makes debugging nearly impossible since behavior becomes dependent on execution order.
Recommended fix: Scope memory structures by runId or clinicId. Use a Map<string, Record<string, string>> keyed by tenant. Clear on run completion.
Type: Architectural Risk

Issue Title: automationService.ts — No Timeout on Full Automation Run, Browser Leak on Crash
Severity: Medium
Category: Architectural Risk
Files: automationService.ts, browser.ts
What is wrong: runAutomation() has no overall timeout. If an action hangs (e.g., waitForSelector with a long timeout, or an approval poll), the run never terminates. The finally { stopAutomationSession } block handles clean shutdown on normal exit, but if the process is killed mid-run, no cleanup occurs. There is also no run-level concurrency cap — N simultaneous runs = N Chrome processes.
Recommended fix: Wrap the entire run in a Promise.race against a maximum run duration timeout (e.g., 20 minutes). Implement a run registry to enforce a configurable maximum concurrent automation sessions per tenant.
Type: Architectural Risk

Issue Title: fhirMapper.ts — AI Confidence Score Written as FHIR Observation Without LOINC Code
Severity: Medium
Category: Misleading Implementation
Files: fhirMapper.ts
What is wrong: AI diagnosis, disposition, and confidence score are mapped to FHIR Observation resources with code: { text: "AI confidence score" } — free text, no LOINC, no SNOMED, no coding system. These observations are non-interoperable and will be rejected or silently dropped by conformant FHIR validators.
Why it matters: Clinically, an AI confidence score being stored as a formal FHIR Observation implies regulatory standing it does not have. Interoperably, downstream systems cannot process uncoded observations. This is labeled as a clinical summary but is structurally invalid.
Recommended fix: Either use appropriate LOINC codes for structured observations (with proper value types), or store AI reasoning as a FHIR DocumentReference or DiagnosticReport.conclusion (text-only, not structured). Do not write AI outputs as clinical Observations without regulatory review.

Issue Title: browser.ts — No Browser Context Isolation, No Credential Sandboxing Per Run
Severity: Medium
Category: Architectural Risk
Files: browser.ts, automationService.ts
What is wrong: Every automation run shares the same Chromium launch configuration with no browser context isolation. Cookies, session storage, and cached credentials from one run can bleed into the next run's page context.
Why it matters: If an Athena session cookie from Clinic A persists in the browser profile, a subsequent Clinic B automation run may interact with Clinic A's session — a cross-tenant EHR access violation.
Recommended fix: Use browser.newContext() per run (Playwright's isolation primitive) rather than browser.newPage(). Dispose the context after each run. Never reuse browser profiles across tenants.
Type: Architectural Risk
