Issue Title: kbConsistencyAudit.ts — SQL joins on non-existent entity_id column
Severity: Critical
Category: Real defect
Files: server/kb/kbConsistencyAudit.ts
What is wrong: Both queries join kb_entity_store on k.entity_id, but the kbEntityStore table in shared/schema.ts has no column named entity_id. The column is id (serial PK) and entity_key (text domain key). The audit SQL will either fail with a column-not-found error on every run, or (if some legacy column exists) join on the wrong field and return meaningless counts.
Why it matters: The KB consistency audit is the mechanism by which operators verify that domain table records (red flags, diagnoses, treatments) are synchronized with the entity store. If it always errors or produces wrong counts, KB corruption goes undetected indefinitely.
Evidence from code: LEFT JOIN kb_entity_store k ON k.entity_type = $1 AND k.entity_id = d.${idCol}::text — entity_id is not a column in kbEntityStore per shared/schema.ts.
Recommended fix: Replace k.entity_id with k.entity_key in both queries. The join should be k.entity_key = d.${idCol}::text where idCol is the domain table's natural key (e.g. rule_id, not id). Update ConsistencyCheck.idColumn to reference the natural key column for each table.
Type: Real defect

Issue Title: kbAuthMiddleware.ts — x-kb-role header accepted as auth
Severity: Critical
Category: Real defect / Security
Files: server/kb/middleware/kbAuthMiddleware.ts, server/kb/routes/kbAdminRoutes.ts
What is wrong: Both requireKbAdmin and requireKbWrite resolve role from req.headers["x-kb-role"] as the first (highest-priority) check, before falling back to req.user?.role. Any client can set this header with no authentication.
Why it matters: KB admin routes control canonical pathway promotion and retirement, which directly determines clinical dispositions. An unauthenticated attacker can promote a pathway that routes all chest pain patients to home care, with no valid session required.
Evidence from code: const role = (req.headers["x-kb-role"] as string | undefined) || (req as any).user?.role. The x-kb-role header check has highest priority.
Recommended fix: Remove the x-kb-role header check entirely. Role must be derived exclusively from a verified JWT or session cookie using requirePhysician / requireProviderSession from the auth layer. Never trust client-supplied role headers.
Type: Real defect

Issue Title: kbRuntime.ts — all loaders return [] on DB failure, silently driving live triage with empty KB
Severity: Critical
Category: Clinical safety / Real defect
Files: server/kb/kbRuntime.ts
What is wrong: loadPriorsFromDb, loadRedFlagsFromDb, and loadTreatmentsFromDb each have a catch block that logs a warning and returns []. When these empty arrays are cached and returned to the triage pipeline: red flag detection finds no rules (no flags triggered for any patient), treatment recommendations return empty, and the Bayesian engine gets no priors (falls back to hardcoded PRIORS table, which may be out of sync with clinical updates).
Why it matters: A DB blip during a busy shift silently serves [] to setRuntimePriors(), which the Bayesian engine logs as a fallback to hardcoded values. An operator with no visibility into the fallback will not know that all patients are being triaged on stale hardcoded logic. A patient with a DB-defined red flag receives no safety escalation.
Evidence from code: } catch (err) { console.warn("[KbRuntime] Failed to load diagnosis priors from DB:", err); return []; } — identical pattern in all three loaders.
Recommended fix: On DB failure: (1) throw a typed KBLoadError rather than returning []; (2) in the cache warm/reload path, catch the error and emit an alert rather than caching the empty result; (3) if a prior cache exists, serve it as a last-known-good value and log a warning that the cache is stale.
Type: Real defect

Issue Title: semanticCache.ts — redis.keys() on every query is a blocking O(n) operation
Severity: Critical
Category: Architectural risk / Performance
Files: server/cache/semanticCache.ts
What is wrong: checkCache() calls redis.keys("rag_cache:*") on every incoming query, then fetches each matching key individually with redis.get(key) in a sequential loop. KEYS in Redis is an O(n) operation that blocks the single-threaded Redis event loop for the duration of the scan.
Why it matters: With 1,000 cached entries, every RAG query triggers a full keyspace scan. In production with many concurrent users, this cascades: queries pile up waiting for Redis, latency spikes from <1ms to seconds, and Redis becomes a bottleneck for all other operations (including KB cache invalidation and escalation counters).
Evidence from code: const keys = await redis.keys("${CACHE_PREFIX}*"); ... for (const key of keys) { const raw = await redis.get(key); ...
Recommended fix: Use Redis HSCAN with a hash structure, or store embeddings in a Redis sorted set with TTL-managed members. For semantic similarity, use a purpose-built vector store (pgvector, Weaviate, Pinecone) rather than iterating all cache entries. At minimum, use SCAN (cursor-based, non-blocking) instead of KEYS.
Type: Architectural risk

Issue Title: kbRuntime.ts reloadAndRewireKbCache — non-atomic reload exposes partial state window
Severity: High
Category: Architectural risk
Files: server/kb/kbRuntime.ts
What is wrong: reloadAndRewireKbCache() calls invalidateKbCache() (sets all three to null) then await Promise.all([loadPriorsFromDb(), loadRedFlagsFromDb(), loadTreatmentsFromDb()]). During the await, any concurrent request calls getKbPriors(), finds _priors === null, calls loadPriorsFromDb() again (a second in-flight load), and the result of whichever load finishes last wins. If loadRedFlagsFromDb fails, _redFlags stays null while the other two are populated — inconsistent state.
Why it matters: During a reload triggered by a KB admin edit, concurrent patient triage requests may see: valid new priors + stale (null) red flags, or valid new red flags + old priors. A patient triaged during this window may receive a disposition based on inconsistent rule versions.
Evidence from code: invalidateKbCache() sets all three to null, then const [priors, redFlags, treatments] = await Promise.all(...) — no lock, no atomic swap.
Recommended fix: Build new cache entries into local variables first, then atomically assign all three: _priors = ...; _redFlags = ...; _treatments = ... in a synchronous block after all three loads succeed. If any fails, restore previous values rather than leaving nulls.

Issue Title: responseCache.ts eviction strategy is FIFO on the insertion-order Map — not LRU
Severity: Medium
Category: Architectural risk
Files: server/cache/responseCache.ts
What is wrong: When the cache is full, set() evicts the first key returned by this.store.keys().next().value — which is the oldest-inserted key, not the least-recently-used key. JavaScript Map preserves insertion order, not access order. Frequently accessed entries that were inserted early are evicted before rarely accessed entries inserted recently.
Why it matters: For a clinical response cache serving repeated queries for common complaints (chest pain, UTI), the most-used entries are most likely to be evicted under load, defeating the cache's purpose and increasing latency for the most common case.
Evidence from code: const firstKey = this.store.keys().next().value; if (firstKey) this.store.delete(firstKey); — this is FIFO, not LRU.
Recommended fix: Either use the existing lru-cache library (already imported in triageCache.ts) consistently for all caches, or implement a proper LRU by tracking access order with a doubly-linked list. Do not use Map for LRU without access-order tracking.
Type: Architectural risk
