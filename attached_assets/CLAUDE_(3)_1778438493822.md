## Project Identity
Auralyn: HIPAA/FDA SaMD Class II urgent care triage platform.
Stack: Node.js + TypeScript monorepo, Express, React 18, PostgreSQL (Drizzle ORM),
Firebase Firestore, BullMQ/Redis, OpenAI GPT-4o, Twilio, shadcn/ui.
Scale: 294 backend routes, 187 frontend pages, 40+ DB tables, 11 BullMQ queues.

## Clinical Domain Vocabulary (use these terms precisely)
- KB = Knowledge Base (PostgreSQL-backed clinical rules)
- SL3-SL8 = Skill Layers (outcome feedback through tenant orchestration)
- RLHF = Reinforcement Learning from Human Feedback (clinical weight updates)
- PHI Guard = applyPHIGuard() wrapper — strips 18 HIPAA identifiers before any LLM call
- Supervisor Gate = Global Safety Gate — fail-closed on DB unavailability
- Golden Cases = ground-truth regression test cases in kb_golden_cases
- Delta Cap = ±2% max RLHF weight change per cycle
- PCCP = Predetermined Change Control Plan (FDA requirement for adaptive algorithms)
- DHF = Design History File (FDA 21 CFR Part 820)
- BAA = Business Associate Agreement (HIPAA vendor contract)
- RLS = Row-Level Security (PostgreSQL, tenant isolation via app.current_tenant_id)

## Routing Map (critical paths — know these before touching anything)
- /api/triage/* → clinical pipeline (intake → scoring → reasoning → disposition)
- /api/safety/* → Safety Veto Agent, red flag detection, PHI guard
- /api/kb/* → Knowledge Base CRUD, entity store, golden cases
- /api/rlhf/* → learning loop, weight proposals, delta cap enforcement
- /api/billing/* → CPT/ICD-10 coding, 837P claim builder, clearinghouse
- /api/fda/* → audit chain, SaMD versioning, 510(k) narrative
- /api/agents/* → agent registry, circuit breakers, evolution proposals
- /api/admin/* → tenant management, user roles, feature flags

## Safety-Critical Files — Extra Caution Required
- server/safety/GlobalSafetyGate.ts — fail-closed logic, do NOT modify without full test run
- server/safety/SupervisorGate.ts — non-bypassable middleware, every route must pass through
- server/agents/SafetyVetoAgent.ts — absolute veto logic, no consensus override
- server/rlhf/DeltaCapGovernor.ts — enforces ±2% cap, 100-outcome minimum
- server/audit/HashChain.ts — SHA-256 immutable chain + S3 sink, do NOT refactor lightly
- server/middleware/tenantContext.ts — sets app.current_tenant_id before every DB query

## Database Rules
- PostgreSQL = system of record for ALL clinical data
- Firestore = conversation state and trace logs ONLY (no clinical writes)
- SQLite = legacy/read-only, NO new clinical writes (write guard enforced)
- Every clinical table has RLS enabled — always set app.current_tenant_id before queries
- Outbox pattern: Firestore writes go through PostgreSQL outbox table first

## BullMQ Queues (never use in-memory for these)
clinical_pipeline, credentialing, claim_submission, triage, notification,
learning, golden-case, auto-healing, audit, ehr-outbound, explanation,
webhook, report, metrics — ALL must use BullMQ/Redis, not in-memory fallback

## Testing Requirements Before Any Clinical Code Change
1. Run golden case regression: npm run test:golden
2. Run scoring instrument unit tests: npm run test:scoring
3. Run safety layer integration tests: npm run test:safety
4. If modifying RLHF weights: run drift detection check: npm run test:drift

## Caveman Sub-Skills for Auralyn

### /caveman-clinical
Terse clinical reasoning responses.
Rules: state the clinical layer affected, the specific file/function,
the safety implication (if any), then the fix. No preamble.
Example output format:
  Layer: Scoring (CURB-65)
  File: server/scoring/CurbScore.ts:47
  Safety: LOW — scoring only, not disposition
  Fix: [code]

### /caveman-hipaa
Terse HIPAA/FDA compliance responses.
Rules: state the regulation (§164.xxx or 21 CFR Part xx),
the specific gap, the required fix, the documentation needed.
No legal hedging unless the answer is genuinely uncertain.
Example output format:
  Reg: HIPAA §164.312(c)(1) — integrity controls
  Gap: SQLite session store has no encryption at rest
  Fix: SQLCipher or deprecate within 90 days
  Doc: Update risk register, section 4.2

### /caveman-agent
Terse agent/RLHF system responses.
Rules: state which agent, which registry, circuit breaker state,
the change, and whether it requires physician review queue.
Example output format:
  Agent: BillingAgent (coordinator registry)
  Breaker: CLOSED (healthy)
  Change: Weight update proposal — requires physician review (confidence 0.73 < 0.85 threshold)
  Action: → kb_evolution_proposals table, status: pending

### /caveman-debug
