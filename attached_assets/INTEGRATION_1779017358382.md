# Auralyn Context Engineering — Integration Guide

This module implements the context engineering principles from the article
**specifically for Auralyn's clinical pipeline**. It addresses four real
failure modes:

1. **Context rot in long encounters** — the model loses red flags buried in
   the middle of a long transcript.
2. **Cross-agent contamination** — every agent reads everyone else's full
   reasoning, paying the cost and inheriting noise.
3. **Tool sprawl** — each agent's action space has gotten too large to
   navigate reliably.
4. **No durable "what we already decided" layer** — the system can
   re-litigate ruled-out diagnoses on a re-run.

---

## What's in this module

| File | What it does |
|------|--------------|
| `types.ts` | The four-tier context model: Immutables, Working, Artifacts, Trace |
| `ClinicalContextManager.ts` | The harness. Assembles every prompt. Single source of truth for encounter state. |
| `ContextCompactor.ts` | Safely reduces working context when it gets long. Promotes droppable info to artifacts before discarding. **Fully deterministic — no model call.** |
| `AgentArtifactBus.ts` | Inter-agent communication via typed artifacts. Enforces producer/consumer contracts. |
| `RoleScopedToolRegistry.ts` | Each agent gets a curated, distinct toolset. Refuses to register two tools that do the same thing for the same role. |
| `ClinicalMemoryStore.ts` | Cross-encounter memory with scope precedence (physician > tenant > global), conflict resolution, and demotion. |
| `example_encounter.ts` | End-to-end wiring example you can run. |
| `index.ts` | Public API — import everything from here. |

## How to drop this into your Replit project

### Step 1 — Copy the files

Drop the `auralyn-context/` folder into your project at `src/context/` (or
wherever your existing clinical pipeline modules live). The files have no
external dependencies beyond `@types/node`, so they should compile alongside
your existing code as-is.

### Step 2 — Tell your AI coder this

Paste this into your Replit assistant chat:

> I have a new module at `src/context/` that implements a tiered context
> model for Auralyn (immutables, working, artifacts, trace). I want you to
> refactor my existing clinical pipeline so every agent step goes through
> `ClinicalContextManager.assemblePromptFor(role, instruction)` instead of
> building prompts ad-hoc. Specifically:
>
> 1. At intake, build the `EncounterContext` with `immutables` populated
>    (chief complaint, vitals, allergies, meds, PMH). Persist
>    `traceRefId` pointing to my S3 audit object.
> 2. Replace every direct LLM call with: assemble prompt via the manager
>    → call model → write outputs back into the manager as `Artifact`s and
>    `bus.publish(role, artifact)`.
> 3. Before each step, call `compactor.shouldCompact(ctx)` and run
>    `compactor.compact(ctx)` if true. Apply the result via
>    `manager.updateWorking(...)` and record the new artifacts.
> 4. Replace the per-agent tool registration with `buildDefaultRegistry()`
>    and pass `tools.toolsFor(role)` to the model call.
> 5. Hook `ClinicalMemoryStore` to my existing Postgres connection by
>    implementing the `MemoryPersistence` interface (table DDL is in the
>    comment at the bottom of `ClinicalMemoryStore.ts`).

### Step 3 — Verify it's working

Run the included example:

```bash
cd src/context
npx tsc && node dist/example_encounter.js
```

You should see output like:

```
[differential] prompt tokens=531, artifacts in=1, tools=2
[billing] artifacts visible=1 (compare with total in encounter: 1)
Action space sizes: { triage: 2, differential: 2, disposition: 4, billing: 2, supervisor: 6 }
```

The two numbers that matter:

- **Prompt tokens per role**: should stay well under the budget defined in
  `PROMPT_BUDGET_TOKENS`. If it doesn't, the compactor needs to be
  triggered more aggressively (lower `workingTokenThreshold`).
- **Artifacts visible to billing**: should be a SMALL fraction of total
  artifacts. That's the whole point. If billing is seeing everything,
  the contracts in `AgentArtifactBus` aren't being honored.

## How each part addresses an article concept

| Article concept | Auralyn implementation |
|---|---|
| **Context offloading** | Trace tier (Tier 4) goes to S3, never inline |
| **Context retrieval** | Artifact-by-role selection in `selectArtifactsForRole` |
| **Context isolation** | Each agent reads only its contracted artifact types |
| **Context reduction** | `ContextCompactor` (deterministic, NOT model-based) |
| **Context rot** | Immutables bookended at top AND bottom of every prompt — middle-of-context is the worst recall zone |
| **Agent harness** | `ClinicalContextManager` IS the harness. All prompt assembly lives here. |
| **Communication via state transfer** | `AgentArtifactBus` with typed contracts |
| **KV cache discipline** | Per-role artifact filtering = stable prompt prefixes per role |
| **Small toolsets** | `RoleScopedToolRegistry` refuses to register colliding capabilities |
| **Agentic memory with revision** | `ClinicalMemoryStore` with conflict resolution + demotion sweep |

## What it does NOT do (deliberately)

- **No model-based compaction.** The article mentions that summarization is
  a common compaction strategy. For clinical use, a summary-by-LLM can
  silently drop a finding. The compactor here is purely rule-based.
- **No automatic red-flag removal.** Once a red flag is in immutables, it
  stays. The supervisor can add hard constraints but cannot remove flags.
- **No PHI in memory.** The memory store is for preferences and learned
  patterns. Patient data lives in the encounter record, never in
  cross-encounter memory.
- **No transcript replay.** Agents do not see prior agents' full reasoning.
  They see distilled artifacts. If you want to debug what an agent
  "thought," that's in the audit trace (S3), not in subsequent prompts.

## How this hooks into work you've already done

You've already shipped several things this module assumes are in place:

- **Row-Level Security**: `ClinicalMemoryStore` honors RLS — see the SQL
  comment in that file. The store passes `tenantId` and `physicianId` into
  every query; your RLS policy should restrict on those.
- **PHI guard hardening**: The artifact bus does not move PHI between
  encounters. Memory entries with PHI should be REJECTED upstream of the
  store (you already have a guard for that).
- **S3 write-once audit sink**: `traceRefId` in `EncounterContext` is the
  pointer to that object. Every step appends to it; the compactor never
  touches it.
- **Supervisor gate**: The supervisor role in `AgentArtifactBus` has the
  broadest consume permissions and exclusive authority to add hard
  constraints via `supervisor_add_hard_constraint`. Wire your existing
  supervisor approval flow to call `manager.addHardConstraint(...)`.

## Next things to add (in priority order)

1. **Postgres implementation of `MemoryPersistence`** — straightforward
   adapter around your existing PG client. DDL is in the comment at the
   bottom of `ClinicalMemoryStore.ts`.
2. **Cron job for `runDemotionSweep`** — daily is fine. Without this,
   unused memory entries accumulate and become noise.
3. **Telemetry on `excluded.artifactIds`** — when the manager drops
   artifacts due to token budget, log it. If you see frequent exclusions,
   you need to either raise the budget for that role or compact more
   aggressively.
4. **Unit tests on `ContextCompactor`** — this is the safety-critical
   piece. The example in `example_encounter.ts` is a starting point; you'll
   want tests covering: red flag preservation, stale differential drop,
   pending question conversion to uncertainty artifacts.
5. **Per-tenant tool overrides** — some tenants will want to add tools or
   restrict them. Currently `buildDefaultRegistry()` returns one global
   registry; you'll want a `buildTenantRegistry(tenantId)` that layers
   tenant-specific tools on top.

## If you're not sure where to start

The single highest-value change you can make today is wiring the
`AgentArtifactBus` between your differential and billing agents. That
alone:

- Cuts the billing agent's prompt size dramatically
- Forces explicit "what was actually decided" artifacts that improve
  documentation
- Reveals where your existing pipeline is implicitly passing reasoning
  forward that shouldn't be (which is itself diagnostic information about
  where regulatory documentation might be weak)
