Issue Title: governanceQueue.ts — Entire governance approval state is in-memory only
Severity: Critical
Category: Architectural risk / Compliance failure
Files: server/governance/governanceQueue.ts
What is wrong: governanceQueue is a module-level array. Every governance item — pending reviews, physician approvals, rejections — lives exclusively in this array. There is no DB write anywhere in the file. updateGovernanceStatus() mutates the in-memory object and returns. On server restart, the entire queue is gone.
Why it matters: A physician approves a high-risk RED_FLAG_RULES change. The server restarts. The approval never happened. The change either gets re-queued as pending or is silently lost. Under HIPAA and FDA 21 CFR Part 11, governance decisions must be durably recorded with the approver's identity, timestamp, and the pre/post state. An in-memory queue satisfies none of these requirements.
Evidence from code: const governanceQueue: GovernanceItem[] = []; — no DB import, no persistence call in any of the four exported functions.
Recommended fix: Replace with a governance_items DB table. Every addGovernanceItem, updateGovernanceStatus call must be an upsert. The in-memory array may remain as a read cache but must never be the source of truth.
Type: Architectural risk + Compliance failure

Issue Title: approvalAudit.ts — Bypasses hash chain, uses raw SQL insert
Severity: Critical
Category: Real defect / Audit integrity
Files: server/audit/approvalAudit.ts
What is wrong: logApproval() inserts directly into audit_logs via db.execute(sql\INSERT INTO audit_logs...`)without callingauditStep(). This bypasses the advisory lock, the queue serialization, and the hash chain entirely. The inserted row has no hashorprevHash. Any subsequent call to verifyEntireChain()orverifyFullAuditChain()will encounter a row with null hashes and either break the chain verification or silently skip it. **Why it matters:** Physician approval events are precisely the events that must be most tamper-evidently recorded. A row inserted outside the hash chain can be deleted or modified without breaking the chain — which is the opposite of the security guarantee the chain is supposed to provide. **Evidence from code:**await db.execute(sql`INSERT INTO audit_logs (trace_id, step, input, output, metadata) VALUES...`)— nocomputeChainHash, no prevHash, no advisory lock. **Recommended fix:** Replace with await auditStep({ traceId: patientId, step: "PHYSICIAN_ACTION", input: { action }, output: overrideData ?? null, metadata: { physicianId, timestamp } })`. Remove the raw SQL insert entirely.
Type: Real defect

Issue Title: modelApproval.ts — applyApprovedUpdate() never actually applies anything
Severity: Critical
Category: Misleading implementation
Files: server/governance/modelApproval.ts
What is wrong: applyApprovedUpdate() calls updateGovernanceStatus(itemId, "approved", reviewedBy) and emits an ENGINE_STATUS event. There is no code that reads the item.change data and updates any model parameter, threshold, weight, or configuration. The governance workflow completes — status becomes "approved", event is emitted, log message printed — but the underlying model or rule is unchanged.
Why it matters: The entire learning loop and model improvement pipeline is predicated on this function actually doing something. If it doesn't, every physician approval is theater: the queue fills up, physicians review and approve, and nothing in the clinical system changes. This is also dangerous in the inverse: if a physician rejects a change, nothing changes either — so the lack of application is invisible.
Evidence from code: const updated = updateGovernanceStatus(itemId, "approved", reviewedBy); — item.change is retrieved but never used to update anything. The function returns updated (a boolean from the status change) not a result of applying the change.
Recommended fix: After updateGovernanceStatus, read item.change, determine the target (e.g. item.sheet), and call the appropriate service to apply the change. This must be audited with auditStep() capturing before/after state.
Type: Misleading implementation

Issue Title: clinicalTrace.ts — traceId collision risk from String(Date.now())
Severity: High
Category: Real defect
Files: server/audit/clinicalTrace.ts
What is wrong: const traceId = String(Date.now()) produces a millisecond Unix timestamp as a string. Two concurrent buildClinicalTrace() calls within the same millisecond — trivially possible in any async environment — produce identical traceIds. The traceHash covers the steps but not the traceId, so the collision is not detected.
Why it matters: FDA export and legal audit both use traceId as the primary lookup key. Colliding traceIds make two distinct patient encounters indistinguishable. In litigation, this breaks the chain of accountability for a specific decision. In audit tooling, one record silently overwrites another.
Evidence from code: const traceId = String(Date.now()); — hashSteps does not include traceId in its hash input, so two records with identical traceIds but different content produce different hashes, making the collision undetectable by verifyTrace().
Recommended fix: Replace with crypto.randomUUID() or a UUIDv4 from the uuid package. If ordering is needed, use \${Date.now()}-\${crypto.randomBytes(4).toString('hex')}.
Type: Real defect

Issue Title: traceStore.ts — In-memory only, 200-trace cap, no persistence
Severity: High
Category: Architectural risk / Compliance failure

Issue Title: auditVerifier.ts — Merkle batch verification is order-inverted and semantically broken
Severity: High
Category: Real defect
Files: server/audit/auditVerifier.ts
What is wrong: verifyAuditBatch() queries orderBy(desc(auditLogs.id)) — newest records first — then computes a Merkle root over those hashes. The audit chain was written in ascending ID order. The Merkle root of a reversed-order batch is a different value than the Merkle root of the same records in chain order. There is also no stored "expected" Merkle root to compare against — the function computes a root and returns it, but never verifies it against anything.
Why it matters: A Merkle root that isn't stored and compared provides zero tamper detection. Computing a root over randomly-ordered records provides zero chain integrity evidence. The function returns verified: hashes.length === records.length — this is true whenever no hashes are null, completely independent of whether any record was tampered with.
Evidence from code: .orderBy(desc(auditLogs.id)) — verified: hashes.length === records.length — no comparison to a stored or expected root.
Recommended fix: Switch to asc(auditLogs.id). Compute the Merkle root and store it in a separate table with the batch range (min_id, max_id, computed_at). On subsequent calls, re-compute and compare against the stored root.
Type: Real defect

Issue Title: requireRole uses "*" as a Permission action — incorrect semantics
Severity: High
Category: Real defect
Files: server/auth/requirePhysician.ts, server/auth/rbacService.ts
What is wrong: In requireRole(): !roles.includes(decoded.role) && !rbacService.can(decoded.role, "*" as any). The intent is "allow admins to bypass role checks." But rbacService.can(role, "*") checks whether ROLE_PERMISSIONS[role].includes("*") — which is true only for admin (whose permissions are ["*"]). For a physician trying to access an admin route that lists them in roles, the first condition !roles.includes("physician") is false (they're included), so they pass correctly. But a physician trying to access a route they're NOT listed in fails the first check, then fails the second check (physicians don't have "*" in their perms). This is correct behavior — but the code comment implies "*" is a wildcard permission check, when it's actually checking if the role itself has admin-level wildcard. The behavior is accidentally correct but semantically wrong and fragile.
Why it matters: A future developer adds "*" to nurse permissions "to simplify things" — suddenly all nurses pass every requireRole check.
Evidence from code: !rbacService.can(decoded.role, "*" as any) — the as any cast is a suppressed TypeScript error revealing the author knew this was wrong.
Recommended fix: Replace with an explicit admin check: const isAdmin = rbacService.can(decoded.role, "tenant:manage") or decoded.role === "admin". Remove the "*" action entirely.
Type: Real defect
