1) Make WhatsApp scenario testing feel like a control panel

Right now you can run scenarios and view traces. The next upgrade is “operator ergonomics.”

A) Add these staff commands (minimal, high value)

!scenario list
returns available JSON testcases + tags

!scenario run <id> [--llm=off|on] [--seed=123]
runs deterministic or LLM-assisted

!trace last
shows last run summary + link to trace viewer

!trace <runId>
fetches and summarizes a specific run

!explain <runId> step <n>
returns only step n: action + inputsUsed + ruleRefs + outputs
(keeps WhatsApp readable)

3) Answering your ChatGPT question: yes, connect it and log it

You can absolutely connect ChatGPT for:

better phrasing

rapport/empathy

handling tangential/problem patients

concise summaries

And yes: you can log it so you can prove it’s better (or not).

The right architecture: “LLM as a tool called by actions”

Add two action types (if you haven’t):

REFRAME_QUESTION(questionId, toneProfile)

DRAFT_SUMMARY(style)

The router decides when to call these actions. The executor performs the LLM call.

Logging: what to store for every LLM call (so it’s measurable)

Create an llm_call_log record per call:

runId, caseId, channel (whatsapp/web)

purpose (“rephrase_question”, “summarize”, “deescalate”)

model, temperature, seed

promptTemplateId (so you know which prompt version)

inputHash + (optionally) redacted prompt content

outputHash + output text

latencyMs, token counts if available

linkedActionStep (trace step number)

Metrics to prove “more engaging / efficient”

Now you can compute:

time-to-answer per question

completion rate of required questions

turns-to-completion

staff escalation rate

patient sentiment (simple classifier or heuristic)

refusal rate / topic drift rate (problem patient signals)

Then run A/B:

same scenarios, LLM off vs on

compare the above metrics

That’s how you turn “it feels nicer” into “it measurably improves completion.”
