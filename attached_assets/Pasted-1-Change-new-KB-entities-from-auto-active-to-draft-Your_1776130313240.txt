2. Align entity-type names between kbTypes.ts and the consistency audit

The audit currently uses values like complaint_pack and red_flag, while kbTypes.ts defines complaint, red_flag_rule, etc. That mismatch means even a fixed join can still report nonsense.

Use one canonical vocabulary everywhere:

// server/kb/kbTypes.ts
export type KbEntityType =
  | "complaint"
  | "question"
  | "modifier"
  | "red_flag_rule"
  | "workup_rule"
  | "diagnosis_rule"
  | "treatment_rule"
  | "disposition_rule"
  | "plan_template"
  | "feature_model"
  | "engine_routing"
  | "clinical_rule"
  | "complaint_pack";

And make the audit use exactly those strings.

3. Do not silently swallow canonical-pathway read failures

The current GET routes can return empty pathway sets during DB failure, which looks like “no data” instead of “broken system.” That is operator-hostile in a safety-critical admin console.

// server/kb/routes/kbAdminRoutes.ts
router.get("/canonical-pathways", async (req, res) => {
  try {
    const { complaintId } = req.query;
    const pathways = await listCanonicalPathways(complaintId as string | undefined);
    res.json({ ok: true, pathways });
  } catch (err: any) {
    res.status(503).json({
      ok: false,
      code: "KB_PATHWAYS_UNAVAILABLE",
      message: err?.message ?? "Canonical pathways unavailable",
    });
  }
});

router.get("/canonical-pathways/:pathwayId", async (req, res) => {
  try {
    const pathway = await getCanonicalPathway(req.params.pathwayId);
    if (!pathway) {
      res.status(404).json({ ok: false, error: "Not found" });
      return;
    }
    res.json({ ok: true, pathway });
  } catch (err: any) {
    res.status(503).json({
      ok: false,
      code: "KB_PATHWAY_UNAVAILABLE",
      message: err?.message ?? "Canonical pathway unavailable",
    });
  }
});
4. Add an actual migration for kb_population_priors

5. Add a startup invariant check

This codebase needs a bouncer at the door. Right now too many important failures become runtime surprises.

// server/startup/assertions.ts
import { getKbCacheStatus, getKbVersion } from "../kb/kbRuntime";

export function assertClinicalStartupInvariants(): void {
  const problems: string[] = [];

  if (typeof getKbVersion !== "function") {
    problems.push("kbRuntime.getKbVersion is missing");
  }

  if (typeof getKbCacheStatus !== "function") {
    problems.push("kbRuntime.getKbCacheStatus is missing");
  }

  if (process.env.NODE_ENV === "production") {
    if (!process.env.REDIS_URL) problems.push("REDIS_URL missing in production");
    if (!process.env.INSTANCE_ID) problems.push("INSTANCE_ID missing in production");
  }

  if (problems.length > 0) {
    throw new Error(`Startup invariant failure:\n- ${problems.join("\n- ")}`);
  }
}

Then call it during server boot.

7. Independent review: encounters.intakeExpiresAt should not be text

That field is carrying time semantics in a string column. That is the kind of tiny schema cut that later bleeds through scheduling, expiration checks, and audit logic. It should be a timestamp.

// shared/schema.ts
intakeExpiresAt: timestamp("intake_expires_at"),

Migration:

ALTER TABLE encounters
  ADD COLUMN intake_expires_at_tmp timestamp;

UPDATE encounters
SET intake_expires_at_tmp = NULLIF(intake_expires_at, '')::timestamp
WHERE intake_expires_at IS NOT NULL;

ALTER TABLE encounters DROP COLUMN intake_expires_at;
ALTER TABLE encounters RENAME COLUMN intake_expires_at_tmp TO intake_expires_at;
8. Independent review: conversationHistory, answers, proposal, physicianSummary should be JSONB

Those fields are currently text fields carrying JSON payloads. That invites malformed data, brittle parsing, and no queryability. In a system obsessed with audit and explainability, those should be first-class JSONB.

9. Independent review: kb_feature_likelihoods.ruleId and similar pseudo-FKs need real constraints

A lot of the schema comments say “FK to…” but the actual foreign key is not enforced. That means the database politely nods while data integrity wanders off into traffic.

10. Independent review: specEngine still needs expression-language hardening

The sandbox is much better than new Function, but long term I would still move from arbitrary JS expressions to a constrained DSL or structured predicates. Sandboxed JS is safer than a bonfire in the server room, but it is still playing with matches.
