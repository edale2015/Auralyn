# ENT Flu Slice - Medical Triage System

## Overview
"env_flu_slice" is an AI-powered medical triage platform for flu-like symptoms, leveraging WhatsApp for initial patient assessments. It aims to provide diagnoses and treatment plans for physician review, automate patient communication, and enhance healthcare efficiency and access. The system is designed for continuous improvement through a self-developing AI architecture, with a vision to transform medical triage into a more efficient, patient-centric process.

## User Preferences
Preferred communication style: Simple, everyday language.

## System Architecture

### Core Architecture
The system employs a constrained agent architecture with a plan/act/observe loop and a multi-system triage pipeline. It features a unified sheets registry for data configuration and diagnostic candidate generation, and a clinical state builder that deterministically assembles an auditable clinical state. A modular, skill-based orchestration layer handles clinical triage, supported by an active control plane for rollouts and rule governance. An intelligence layer provides explainability and failure-driven rule suggestions, with an extended learning loop for continuous improvement. The Clinical Intelligence Planning Layer (CIPL) automatically identifies areas for improvement, resolves knowledge gaps, detects model drift, and prioritizes enhancements. A Clinical Governance & Deployment Layer ensures the safety of clinical changes before deployment. The entire clinical AI system follows a 12-layer architecture encompassing Interface, Normalization, State, Knowledge, Safety, Reasoning, Decision, Learning, Analytics, Governance, Integration, and Orchestration.

### UI/UX Decisions
The frontend is built with React 18, TypeScript, `shadcn/ui`, and Tailwind CSS, offering intuitive interfaces for physicians, patients, and administrators. Key interfaces include physician login, patient intake, case status, visit summaries, physician dashboard, administrative consoles, and specialized dashboards like the Clinical Simulation Lab, Clinical Control Tower, and Executive Dashboard for advanced system insights and management.

### Security & Quality Hardening (Latest)
- **bcrypt Password Hashing**: All demo user passwords are now hashed with bcrypt (12 rounds) at startup. Plain-text password comparisons fully eliminated. Passwords can be overridden per-role via `DEMO_PASSWORD_<ROLE>` environment variables.
- **JWT Security**: `APP_JWT_SECRET` now throws a fatal error at startup in production if not set. Access tokens reduced to 15-minute expiry. Refresh token endpoint added at `POST /api/roleAuth/refresh` (7-day expiry).
- **Route Authentication**: `/smart-intake/review-queue`, `/all-cases`, `/audit-log`, `/case/:id`, `/batch-approve`, `/approve-all-safe`, `/outcomes`, `/outcomes/analytics` all now require valid JWT Bearer tokens with appropriate role check (`clinicianOnly` or `adminOrPhysician`).
- **Tenant Guard Hardened**: `tenantGuard` middleware now cross-checks requested `clinicId` against the authenticated user's `organizationId` — non-admin users cannot access other clinic's data by spoofing headers.
- **Rate Limiting**: `express-rate-limit` applied to `/api/clinical/*` and `/api/patients/*` (30 req/min per IP). Auth endpoints (`/api/roleAuth/login`) rate-limited to 20 attempts per 15 minutes. Webhook limiter (100/min) available for Twilio routes.
- **Orchestrator: No More Silent Failures**: `notifyOnCallPhysician` failure now emits a `CRITICAL` alert event to the Control Tower and logs to the console. Learning cycle errors emit `ERROR` events. All `.catch(() => {})` replaced with meaningful error handling.
- **Autonomy: Uncertainty Defaults to 0.5**: Orchestrator now passes `uncertainty: scores?.uncertainty ?? 0.5` — defaulting to "uncertain" rather than "certain", preventing incorrect AUTO decisions when the engine doesn't return an uncertainty score.
- **Bounded In-Memory Stores**: `fixStore` (autoFixEngine) and `feedbackStore` (feedbackEngine) both capped at 1000 entries with oldest eviction — no more unbounded memory growth.
- **DB Health Check**: `GET /api/monitoring/health` now runs `SELECT 1` against Postgres and returns `database: pass/fail` in the response — visible to load balancer health probes.
- **PHI Sanitizer** (`server/utils/phiSanitizer.ts`): `sanitizeForLog(obj)` recursively redacts 20+ PHI field names before any object reaches logs. `redactPhi(obj, fields)` for targeted redaction.
- **Structured Logger** (`server/utils/logger.ts`): `logger.info/warn/error/debug()` — emits JSON in production (CloudWatch-friendly), readable text in development.
- **Pagination on Intake Routes**: All `/smart-intake` list endpoints now accept `?limit=` and `?offset=` query params (max 200/500). Returns `{ total, offset, limit, cases/entries }` shape.
- **Unit Tests** (vitest): 33 unit tests across 4 files: `safetyGate.test.ts` (10 tests), `autonomyEngine.test.ts` (7 tests), `confidenceCalibration.test.ts` (8 tests), `phiSanitizer.test.ts` (8 tests). Run with `npx vitest run --config vitest.config.ts`.
